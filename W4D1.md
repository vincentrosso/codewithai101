# Week 4, Day 1 — Meet Claude Code
*v1.0.0*

**Goal for today:** install Claude Code, point it at brand-lens, and take a read-only tour. Three new rules — review every diff, never auto-approve a command, ask before committing unclear code — replace Week 2's two rules. By end of day Claude Code has explained your repo to you and you haven't accepted a single edit yet.

**Time budget:** ~75 min hands-on, ~15 min journal, ~30 min slack.

---

## About the tool — what changes today

You met Claude in a browser tab on W2D1. That was a chat: paste code, read explanation, type code yourself. Claude Code is the same Claude — but it runs in your terminal, can read files in your repo, can edit them, and can run commands. The chat is now a colleague who has hands.

That power needs new rules. Week 2 had two:

1. Ask Claude to explain, not to write.
2. Run everything yourself.

Week 4 has three:

**Rule 1: Review every diff before accepting.** Claude Code will propose changes as diffs (added lines marked `+`, removed lines marked `-`). Read every line. If you accept a diff you didn't read, you don't own the code — and you can't debug it later.

**Rule 2: Never auto-approve a command you can't explain.** Claude Code asks before running shell commands. The temptation to hit "yes" is real. Don't. If you don't know what `rm -rf node_modules` does, say no, ask why it wants to run that, then decide.

**Rule 3: If a line is unclear, ask before committing.** When Claude Code edits a line you don't fully understand, ask Claude Code to explain *that line* before you commit. Don't commit code you can't read.

These rules tighten as you get more access, not loosen.

## 1. Warm-up (~5 min)

Activate venv. From inside `~/dev/brand-lens`:

```
source .venv/bin/activate
python3 runner.py --limit 1
```

Confirm Week 3's pipeline still works. You'll need this baseline tomorrow.

## 2. Install Node, then Claude Code (~20 min)

Claude Code installs via `npm`, which comes with Node.js. Install Node via Homebrew:

```
brew install node
```

This is the same pattern as W1D2 — Homebrew installs the tool, then the tool's package manager (`npm`) installs the thing you want. Verify:

```
node --version
npm --version
```

Both should print version numbers. Now install Claude Code:

```
npm install -g @anthropic-ai/claude-code
```

The `-g` flag installs it globally so the `claude` command is available from any directory. When it finishes:

```
claude --version
```

Should print a version number. If you see `command not found`, your shell hasn't picked up the new binary — open a new terminal tab and try again.

## 3. First session — log in (~10 min)

In your terminal:

```
cd ~/dev/brand-lens
claude
```

The first time you run `claude` it walks you through authentication. Follow the prompts (it'll open a browser, you log in with the same account you used for claude.ai on W2D1, then come back to the terminal).

Once logged in, you're in an interactive session inside `~/dev/brand-lens`. Claude Code can read any file in this directory tree. It can also edit files and run commands — but only after asking you.

## 4. Read-only tour: ask it to summarize (~15 min)

You're going to use Claude Code as a reader, not a writer, today. Type:

> Summarize the structure of this repo. What are the main files, and what does each one do? Don't make any edits.

Read the response carefully. Compare it against what you know is true:

- Are the file descriptions accurate?
- Did it claim a file does something it doesn't?
- Did it miss a file or invent one?

This is the same calibration exercise as W2D1's first Claude session — except now Claude Code is reading the actual files, not what you pasted. Its accuracy will be different.

## 5. Ask about a specific function (~15 min)

Pick one function from `runner.py`, `fetch.py`, or `parse.py` — ideally the one you struggled to explain in W3D5 prep. Ask:

> Walk through `process_url` (or whatever you picked) line by line. Treat me like someone four weeks into Python.

Read the explanation. For every line:

- Does the explanation match what you understand?
- Is there a line where Claude Code's explanation makes more sense than yours?
- Is there a line where Claude Code is confidently wrong?

## 6. Notice the temptation to skim a diff (~10 min)

Now ask Claude Code to *propose* a small edit — but don't accept it:

> Propose a small improvement to `runner.py` — show me the diff but don't apply it.

When the diff appears, watch what you feel. Is there a pull to skim and say yes? That pull is real. Notice it now, while there's no pressure.

Read every `+` line. If anything is unclear, ask:

> Explain line N of that diff in plain language.

Then **reject the change** for today. Type `n` or whatever rejects the proposed edit. You can revisit it tomorrow.

## 7. Exit Claude Code, commit nothing (~5 min)

Type `/exit` or Ctrl-D to leave the session. There should be nothing to commit — you didn't accept any edits. Run `git status` to confirm:

```
git status
```

Should say "nothing to commit, working tree clean."

`journal/W4D1.md`. Use this structure:

## What "done" looks like for W4D1

- `claude --version` prints a version number
- A `claude` session ran inside `~/dev/brand-lens`
- Claude Code summarized your repo and walked through one function
- You proposed an edit, read the diff, and rejected it
- `git status` is clean — no edits accepted today
- A W4D1 journal exists naming one place Claude Code was wrong and one moment Rule 1 felt hard to follow
