## 2. Test render_brief — the easy case (~25 min)

`render.py` is now part of the MVP, so it needs tests. But notice the contrast with the last two weeks before you write a single line:

- `fetch_one` (W5D4) needed `monkeypatch` and a `FakeResponse` — it touches the network.
- `summarize_brand` (W7D4) needed `monkeypatch` and a `FakeMessage` — it calls a paid API.
- `render_brief` needs **none of that.** Hand it a context dict, it returns a string. Same input, same output, no network, no disk, no key. It's *pure* (W5D3) — and pure functions are the easiest thing in the world to test. This is the foundations day's "push side effects to the edges" paying off in your own code: you split `render_brief` (pure) from `write_brief` (side effect) on D1 precisely so today would be this simple.

Create `tests/test_render.py`:

```python
from render import render_brief


def make_context(brief=None, pages=None):
    return {
        "brand_name": "Innisfree",
        "generated_at": "2026-05-27",
        "brief": brief,
        "pages": pages or [],
    }


def test_render_brief_includes_brand_name():
    output = render_brief(make_context())
    assert "Innisfree" in output


def test_render_brief_shows_summary_when_brief_present():
    brief = {
        "value_proposition": "Clean beauty from Jeju Island",
        "messaging_consistency": "consistent",
        "us_presence_signal": "canonical points to US site",
        "confidence": "medium",
    }
    output = render_brief(make_context(brief=brief))
    assert "Clean beauty from Jeju Island" in output
    assert "medium" in output


def test_render_brief_shows_fallback_when_no_brief():
    output = render_brief(make_context(brief=None))
    assert "No brief" in output


def test_render_brief_lists_pages():
    pages = [{"url": "https://www.innisfree.com/", "title": "Innisfree Home"}]
    output = render_brief(make_context(pages=pages))
    assert "Innisfree Home" in output
    assert "https://www.innisfree.com/" in output
```

What's being checked, and the new idea:

- **`make_context` is a helper, not a fixture.** It's a plain function that builds a context with sensible defaults, letting each test override just the part it cares about (`brief=...` or `pages=...`). You *could* make it a fixture; a helper is fine when you need to vary the input per test.
- **`assert "..." in output`** — these are **substring** assertions. The rendered markdown is a big string; each test checks that the right piece *appears in* it. You're not asserting the entire output character-for-character (that would break every time you tweaked a heading) — you're asserting the *behavior*: the brand name shows up, a present brief renders its fields, an absent brief renders the fallback, pages are listed. That's testing behavior, not implementation (W5D5).
- **The fallback test** pins the `{% else %}` branch from D1 — if someone deletes it, the "No brief" text vanishes and this test fails.

Run them:

```
pytest -v
```

All your earlier tests plus four new `test_render` tests, green. Note how fast that was to write compared to the mocking days — that speed *is* the reward for keeping `render_brief` pure.

## 3. Pin requirements.txt (~15 min)

Your project needs a recipe so it installs the same way anywhere (W8D3 §4). Find the versions you actually have installed:

```
pip freeze | grep -i -E "requests|beautifulsoup4|pyyaml|pytest|anthropic|python-dotenv|jinja2"
```

That prints each package with its exact version, like `jinja2==3.1.4`. Now write `requirements.txt` with those lines — **only the libraries your project imports**, not everything `pip freeze` would dump:

```
touch requirements.txt
subl requirements.txt
```

```
requests==2.32.3
beautifulsoup4==4.12.3
pyyaml==6.0.2
anthropic==0.40.0
python-dotenv==1.0.1
jinja2==3.1.4
pytest==8.3.4
```

(Use *your* actual versions from the `pip freeze` output — the numbers above are examples.) Each `==` pins an exact version so the project is reproducible (W8D3 §4). Notice `pytest` is here too: it's a dependency for *developing* the project even though the pipeline doesn't import it. (Bigger projects split these into a separate dev-requirements file; one file is fine for now.)

> Why not just `pip freeze > requirements.txt`? Because that captures every package in your venv, including ones pulled in indirectly that you never chose. The hand-written list says what the project *means* to depend on. The freeze output is a starting point to copy versions from, not the final file.

```markdown
# Brand Lens

Brand Lens evaluates how a Korean brand presents itself to the US market.
It fetches a brand's US web pages, extracts the fields that signal positioning
(title, meta description, og:title, h1, canonical URL), stores them in a SQLite
database, asks Claude to turn them into a structured brief, and renders one
markdown brief per brand.

## Setup

    git clone <your repo url>
    cd brand-lens
    python3 -m venv .venv
    source .venv/bin/activate
    pip install -r requirements.txt
    cp .env.example .env          # then add your ANTHROPIC_API_KEY

## Usage

    python3 runner.py             # fetch + parse + store pages in the database
    python3 summarize_all.py      # generate a brief per brand (calls the LLM)
    python3 generate_briefs.py    # render briefs/<brand>.md from the database

Briefs are written to briefs/ (gitignored — regenerate any time).

## Layout

- runner.py      — orchestrates fetch → parse → store
- fetch.py       — fetch a page (network)
- parse.py       — HTML → fields
- db.py          — all SQLite access
- brief.py       — LLM summarization
- render.py      — context → markdown
- generate_briefs.py — db → rendered briefs/
- templates/     — jinja2 templates
- tests/         — pytest suite
```

Adjust the commands to match what your scripts are actually called. **The test of a good README** (W8D3 §5): could someone clone this and get a brief out *without asking you*? Read yours and check that the path from "clone" to "a brief appears" is complete — no missing step.
