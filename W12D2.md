# Week 12, Day 2 — The Guardrail: Only Ship Green
*v1.0.0*

**Goal for today:** close the last gap. Right now the deploy runs on *every* merge to `main` — even one that somehow broke the tests. Today you wire the deploy to run **only if the tests pass**, so "we only ever publish working code" becomes a rule the machinery enforces, not a hope. You'll make the deploy job *depend on* the test job, then prove it by trying to ship a red change and watching the deploy refuse.

**Time budget:** ~80 min hands-on, ~15 min journal, ~15 min slack.

---

## 1. Warm-up (~5 min)

```
source .venv/bin/activate
git switch main && git pull
pytest -v
```

Green. Yesterday you made `main` deploy itself. Today you make sure it only deploys when it *should*.

## 2. The gap, named (~10 min)

Walk the current setup honestly. You have two separate workflows:

- `ci.yml` — runs `pytest` on every push and PR.
- `deploy.yml` — runs the S3 sync on every push to `main`.

They don't know about each other. So picture the hole: a change merges to `main`, the deploy workflow fires *immediately* and ships the site — **at the same time** the CI workflow is independently discovering the tests are red. The deploy doesn't wait for, or care about, the test result. Branch protection (W10D5) blocks *merging* a red PR, which helps — but it's a belt without suspenders. The cleaner, more honest design is: **the deploy itself should refuse to run unless the tests have passed.** Make the publish *structurally* conditional on green.

## 3. One workflow, two jobs, a dependency (~25 min)

The fix is to put the test and the deploy in *one* workflow as two jobs, and declare that deploy **needs** test. A job that `needs` another waits for it and runs only if it *succeeded*. If the test job fails, the deploy job is skipped entirely — nothing ships.

Combine your two workflows into one. Replace `deploy.yml` with a single pipeline (and delete the now-redundant standalone deploy):

```
subl .github/workflows/deploy.yml
```

```yaml
name: Test and Deploy

on:
  push:
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install -r requirements.txt
      - run: pytest -v

  deploy:
    needs: test                                    # <-- the guardrail
    if: github.ref == 'refs/heads/main'            # <-- only on main
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install -r requirements.txt
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1
      - run: python3 build_site.py
      - run: aws s3 sync site/ s3://brand-lens-tyler-2026 --delete
```

The two lines that do the work:

- **`needs: test`** — deploy waits for `test` and runs *only if it passed*. Red tests ⇒ skipped deploy. This is the guardrail: the publish can't happen without a green suite ahead of it.
- **`if: github.ref == 'refs/heads/main'`** — deploy only on the `main` branch. Now this *one* workflow runs `test` everywhere (so PRs still get checked) but only deploys from `main`. Test-on-PR and deploy-on-main, unified, with the dependency between them explicit.

Since `test` now lives here and runs on every push/PR, you can delete the old standalone `ci.yml` to avoid running the suite twice:

```
git rm .github/workflows/ci.yml
```

(Then, if your branch-protection rule named the check `CI / test`, update it under **Settings → Branches** to the new job's check name, `Test and Deploy / test`, so the required-check rule still points at a real check.)

## 4. Ship it (~10 min)

```
git switch -c gate-the-deploy
git add .github/workflows/
git commit -m "Gate deploy on passing tests: deploy needs test, deploy only on main"
git push -u origin gate-the-deploy
```

Open the PR. The `test` job runs (green). Merge. In **Actions**, watch the combined workflow: `test` runs first, and *because it passed*, `deploy` runs after it and ships. Same outcome as yesterday — but now the deploy ran *because* the tests were green, not merely alongside them.

## 5. Prove the guardrail — try to ship a red change (~15 min)

A guardrail you've never seen stop anything is decoration. Try to deploy broken code and watch the deploy refuse.

```
git switch -c try-to-ship-broken
```

Break a test (W10D4 §6 — a wrong assertion in `tests/test_build_site.py`) *and* make a visible site change (a new heading), so you'd *see* it if it shipped. Commit, push, open the PR.

On the PR, the `test` check goes **red** — and branch protection won't let you merge. To *prove the deploy guardrail itself* (not just branch protection), the cleanest demonstration: notice that even if the change reached `main`, the `deploy` job's `needs: test` would skip it. You can see this logic on the PR's Actions run — `test` failed, `deploy` shows **skipped**, grey, never ran. The visible heading change is *not* on the live site. **Broken code shipped nothing.**

Throw it away:

```
git switch main
git push origin --delete try-to-ship-broken
git branch -D try-to-ship-broken
```

Confirm the live site is unchanged and `main` is green. The guardrail held: the only way to the live site is *through* green tests.

## 6. Optional — a scheduled rebuild (~5 min, read now, do if time)

One more thing a real deploy often wants: a *scheduled* run, so the site refreshes on a cadence even when nobody pushes. GitHub Actions can trigger on a clock, the same `cron` idea as a server's daily job. You'd add to the `on:` block:

```yaml
on:
  push:
  pull_request:
  schedule:
    - cron: "0 12 * * 1"      # every Monday at 12:00 UTC
```

This is the same shape as autoarb's daily cron, hosted by GitHub instead of a server. For Brand Lens, a real schedule would also need to *re-run the pipeline* (fetch + summarize) before building — which costs LLM money, so it's a deliberate choice, not a default. Know the mechanism exists; wiring the full scheduled pipeline is a fine stretch goal, not required.
