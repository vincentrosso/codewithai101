## 2. The two ways yesterday's code breaks (~10 min)

`summarize_brand` from D2 works when everything goes right. Two things go wrong in real use, and right now either one *crashes the whole program*:

1. **The API call fails.** Rate limit hit (W7D3 §4), network blip, bad key, Anthropic having a bad day. `client.messages.create` raises, and an unhandled exception kills the pipeline mid-run.
2. **The reply won't parse.** Claude wrapped the JSON in prose or fences (you saw this on D2 §7). `json.loads` raises `JSONDecodeError`, same fatal result.

You've solved this exact shape before. `fetch_one` (W1D4) wraps a flaky network call in `try/except` and **never raises** — it returns `(status, html, error)`, putting the failure in the return value instead of crashing. `summarize_brand` needs the same contract: one bad brand shouldn't take down the run for all the others.

## 3. Make summarize_brand robust (~20 min)

Rewrite `summarize_brand` in `brief.py` so it returns a `(summary, error)` tuple and never raises — the same shape as `fetch_one`:

```python
def summarize_brand(brand_name, pages):
    """Return (summary_dict, error). On any failure, summary is None and
    error is a string. Never raises."""
    prompt = build_prompt(brand_name, pages)

    try:
        message = client.messages.create(
            model="claude-haiku-4-5",
            max_tokens=500,
            system=SYSTEM,
            messages=[{"role": "user", "content": prompt}],
        )
    except Exception as e:
        return None, f"api call failed: {e}"

    raw = message.content[0].text
    try:
        summary = json.loads(raw)
    except json.JSONDecodeError:
        return None, f"could not parse JSON from reply: {raw[:200]}"

    return summary, None
```

Read the two `try/except` blocks — each guards one of the failure modes from section 2:

- The **first** wraps the network call. If Claude's server errors or the network dies, you get `(None, "api call failed: ...")` instead of a crash.
- The **second** wraps the parse. If the reply isn't clean JSON, you get `(None, "could not parse JSON: ...")` — and the error string includes the first 200 characters of what Claude *actually* sent, so you can see what went wrong.

This is W1D4's lesson reused: push the failure into the return value so the *caller* decides what to do with a bad brand, instead of the whole run dying. The contract is now "summarize_brand always returns a tuple; check the error."

> This is a real change to a function's return type. If you used Claude Code to make this edit, review every line of the diff (W4D3) and update any caller — yesterday's `try_brief.py` unpacked one value; it now unpacks two.

## 4. Store the brief in the database (~15 min)

The brief is data, and data goes in the database (Week 6). Add a column to hold it.

First, teach the schema about it. In `db.py`, add `brief TEXT` to the `brands` table in `SCHEMA`:

```python
CREATE TABLE IF NOT EXISTS brands (
    slug  TEXT PRIMARY KEY,
    name  TEXT,
    brief TEXT
);
```

`IF NOT EXISTS` means this won't alter a table that already exists — so your existing `brand_lens.db` won't gain the column automatically. Add it once with `ALTER TABLE` (this is how schemas evolve in the real world):

```
sqlite3 brand_lens.db "ALTER TABLE brands ADD COLUMN brief TEXT;"
```

Then a helper to write a brief, in `db.py`:

```python
def set_brief(conn, brand_slug, summary):
    conn.execute(
        "UPDATE brands SET brief = ? WHERE slug = ?",
        (json.dumps(summary), brand_slug),
    )
```

Two things:

- **`UPDATE ... WHERE`** changes an existing row instead of inserting a new one — the brand row already exists; you're filling in its `brief`. The `WHERE slug = ?` is what stops it from overwriting *every* brand's brief (leave the `WHERE` off and you'd set them all — a classic, painful SQL mistake).
- **`json.dumps(summary)`** — `summary` is a dict; a database column holds text. `json.dumps` turns the dict into a JSON string to store. (To read it back you'd `json.loads` it — the round trip you know from Week 1.) Make sure `import json` is at the top of `db.py`.

## 5. Wire it together and run it once (~10 min)

A small script that does the full loop — read brand data from the DB, summarize, store the brief back:

```
touch summarize_all.py
subl summarize_all.py
```

```python
import db
import brief

conn = db.get_connection()

for slug, name in conn.execute("SELECT slug, name FROM brands").fetchall():
    pages = db.page_fields_for_brand(conn, slug)
    summary, error = brief.summarize_brand(name, pages)
    if error:
        print(f"{slug}: SKIPPED — {error}")
        continue
    db.set_brief(conn, slug, summary)
    print(f"{slug}: brief stored")

conn.commit()
conn.close()
```

Notice the payoff of the `(summary, error)` contract: when one brand fails, the script prints why and **continues** to the next brand (`continue`). No brand's failure stops the others. That's the robustness from section 3, in action.

```

```

Then confirm a brief landed:


You should see the JSON brief stored against each brand. The full Brand Lens loop now exists: fetch → parse → store → summarize → store the judgment.

## 6. Test it — by mocking Claude (~30 min)

`brief.py` is now load-bearing and Claude Code will edit it. It needs tests. But it has the worst possible side effect to test against — a slow, non-deterministic, *paid* network call (W7D3 §6). You will **not** call the real API in a test. You'll mock it, exactly like you mocked `requests.get` in W5D4.

Recall the W5D4 pattern: a fake object with the same *shape* as the real thing, plus `monkeypatch` to swap it in for the duration of the test. The real reply is a `message` object where `message.content[0].text` holds the text. So the fake needs that shape:

Create `tests/test_brief.py`:

```python
import json

import brief


class FakeBlock:
    def __init__(self, text):
        self.text = text


class FakeMessage:
    def __init__(self, text):
        self.content = [FakeBlock(text)]


def test_summarize_brand_parses_json_reply(monkeypatch):
    payload = {
        "value_proposition": "Clean beauty from Jeju",
        "messaging_consistency": "consistent",
        "us_presence_signal": "canonical points to US site",
        "confidence": "medium",
    }

    def fake_create(**kwargs):
        return FakeMessage(json.dumps(payload))

    monkeypatch.setattr(brief.client.messages, "create", fake_create)

    summary, error = brief.summarize_brand("Innisfree", [])

    assert error is None
    assert summary["confidence"] == "medium"
    assert summary["value_proposition"] == "Clean beauty from Jeju"


def test_summarize_brand_returns_error_on_bad_json(monkeypatch):
    def fake_create(**kwargs):
        return FakeMessage("Here is your analysis: {not valid json")

    monkeypatch.setattr(brief.client.messages, "create", fake_create)

    summary, error = brief.summarize_brand("Innisfree", [])

    assert summary is None
    assert "parse" in error


def test_summarize_brand_returns_error_when_api_raises(monkeypatch):
    def fake_create(**kwargs):
        raise RuntimeError("rate limited")

    monkeypatch.setattr(brief.client.messages, "create", fake_create)

    summary, error = brief.summarize_brand("Innisfree", [])

    assert summary is None
    assert "api call failed" in error
```

Walk through what's mocked and why, mapping each piece to W5D4:

- **`FakeMessage` / `FakeBlock`** — the minimum-viable fake. The real reply has lots of attributes; your code only reads `.content[0].text`, so the fake only has *that*. (W5D4's `FakeResponse` only had `status_code` and `text` for the same reason.)
- **`def fake_create(**kwargs)`** — stands in for `client.messages.create`. It takes `**kwargs` so it swallows whatever arguments `summarize_brand` passes (`model`, `max_tokens`, `system`, `messages`) without caring about them — its job is only to return a canned reply. (W5D4's `fake_get` had to accept `url` and `timeout` for the same reason.)
- **`monkeypatch.setattr(brief.client.messages, "create", fake_create)`** — the swap. After this line, `summarize_brand`'s call to `client.messages.create` lands in your fake. No network, no key spend, instant. `monkeypatch` puts the real one back when the test ends.
- **The three tests** map exactly to the three outcomes: clean JSON → `(dict, None)`; junk reply → `(None, parse error)`; the API raising → `(None, api error)`. The last two are the failure paths you built in section 3 — and like the error-path test in W5D4, they verify behavior you *cannot trigger reliably in real life* (you can't make Anthropic rate-limit you on demand). Mocking is the only way to test them.

Run it:

```

```

Your Week 5 + Week 6 tests plus three new `test_brief` tests, all green, still under a second — no API calls, nothing billed.

> One honest note: importing `brief.py` constructs `Anthropic()`, which reads your key from the environment (`.env`). So the tests need the key *present* to build the client — but they never *call* the real API, because `create` is mocked. The key is read; no request is sent; nothing is charged.
