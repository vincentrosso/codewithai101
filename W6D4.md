## 2. The re-run problem (~10 min)

Your pipeline runs every day. Day one, it fetches `https://www.innisfree.com/` and stores a row. Day two, it fetches the same URL again. What should happen?

You declared `url` as `UNIQUE` in the schema (D1). So a second plain `INSERT` of the same URL is *supposed* to fail. Let's confirm. **Predict first** — write in your journal what you think happens — then run:

```
python3 -c "import db; conn = db.get_connection(); db.insert_brand(conn, 'innisfree', 'Innisfree'); db.insert_page(conn, 'innisfree', {'url': 'https://www.innisfree.com/', 'fetched_at': '2026-05-28T10:00:00'}); conn.commit()"
```

If `https://www.innisfree.com/` is already in your table from yesterday, you get:

```
sqlite3.IntegrityError: UNIQUE constraint failed: pages.url
```

That's the `UNIQUE` constraint from D1 doing exactly its job — refusing a duplicate URL. (If your prediction was "it adds a second row," edit your journal: the constraint is what makes that impossible, and that's the behavior you *want*.) But "crash on every re-run" isn't usable. You need "insert if new, update if seen before." That operation has a name: **upsert** (update-or-insert).

## 3. upsert_page — insert-or-update (~15 min)

Add to `db.py`:

```python
from datetime import datetime


def upsert_page(conn, brand_slug, record):
    conn.execute(
        """
        INSERT OR REPLACE INTO pages
            (brand_slug, url, status, title, description,
             og_title, h1, canonical, error, fetched_at)
        VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
        """,
        (
            brand_slug,
            record["url"],
            record.get("status"),
            record.get("title"),
            record.get("description"),
            record.get("og_title"),
            record.get("h1"),
            record.get("canonical"),
            record.get("error"),
            record.get("fetched_at") or datetime.now().isoformat(),
        ),
    )
```

Two things changed from `insert_page`:

- **`INSERT OR REPLACE`** — if a row with this `url` already exists (the `UNIQUE` column), SQLite deletes the old row and inserts the new one. Run it ten times for the same URL, end with one row, holding the latest data. (Honest caveat: `REPLACE` gives the row a fresh `id` each time, because it's really a delete-then-insert. Nothing in brand-lens points at a page's `id`, so it doesn't bite us — but it's the kind of detail worth knowing exists.)
- **`record.get("fetched_at") or datetime.now().isoformat()`** — `fetched_at` is `NOT NULL` in the schema, and a record coming straight from the pipeline won't have it set. So if it's missing, stamp it with the current time. `datetime.now().isoformat()` produces a string like `2026-05-27T14:30:00`.

Run the re-fetch from section 2 again — twice — and confirm it no longer errors and doesn't duplicate:

```
python3 -c "import db; conn = db.get_connection(); db.upsert_page(conn, 'innisfree', {'url': 'https://www.innisfree.com/', 'title': 'updated title'}); conn.commit()"
sqlite3 brand_lens.db "SELECT COUNT(*) FROM pages WHERE url = 'https://www.innisfree.com/';"
```

The count is `1`, no matter how many times you run the upsert. That's idempotence — running it again doesn't change the outcome.

## 4. Wire it into runner.py (~25 min)

Now connect it to the pipeline. Your `main()` already loops over brands and their pages, building records. You're going to write each record to the database in that same loop.

This is real integration work in *your* `runner.py`, whose exact shape is yours. **AI is allowed** — Week 4 rules apply: review every line of any diff, ask Claude Code to explain anything you don't follow, don't commit code you can't read. A good instruction to Claude Code is specific about the behavior you want:

> In runner.py, add a `--db` flag (default `brand_lens.db`). At the start of main(), call `db.init_db(args.db)`. In the loop where each page record is built, also write it to the database: open one connection for the run, call `db.insert_brand(conn, slug, name)` once per brand and `db.upsert_page(conn, slug, record)` per page, commit once at the end, and close. Keep the existing JSON/CSV output working unchanged. Show me the diff before applying.

The shape you're aiming for, sketched (yours will differ in detail):

```python
import db

# ... in main(), once, before the brand loop:
db.init_db(args.db)
conn = db.get_connection(args.db)

# ... inside the loop, for each brand slug + name:
db.insert_brand(conn, slug, name)
#     and for each page record under that brand:
db.upsert_page(conn, slug, record)

# ... after the loops:
conn.commit()
conn.close()
```

Notice the connection lifecycle: open **once** per run, write many rows, **commit once**, close. Opening and committing per row would work but be needlessly slow — like saving a document after every keystroke.

When the diff looks right and you understand every line, run the pipeline:

```
python3 runner.py --limit 2
```

Then look at what landed:

```
sqlite3 brand_lens.db "SELECT brand_slug, url, status FROM pages ORDER BY brand_slug;"
```

Run the pipeline a **second** time and check the count didn't grow — the upsert means a re-run refreshes rows instead of piling up duplicates. This is the payoff of the `UNIQUE` constraint you set on D1 and flagged on D3.

## 5. Test the db layer — with an in-memory database (~25 min)

`db.py` is now load-bearing code Claude Code will edit later. Per the whole point of Week 5 — and the [why-test](sidelines/why-test.md) sideline — it needs tests. But there's a problem you already know how to think about: these functions have a **side effect** (they write to a database file). W5D3 taught that side-effectful code is harder to test, and W5D4 taught the fix — substitute a fake for the real thing.

Here the fake is beautifully simple. SQLite can create a database that lives **only in memory**, never touching disk, by connecting to the special name `":memory:"`. It's a real database with real SQL — it just evaporates when the connection closes. Perfect for a test: fast, isolated, and it can't pollute your real `brand_lens.db`.

Add a fixture to `tests/conftest.py` (the same file your `full_page_html` fixture lives in):

```python
import sqlite3
import db


@pytest.fixture
def db_conn():
    conn = sqlite3.connect(":memory:")
    conn.executescript(db.SCHEMA)
    yield conn
    conn.close()
```

This is a fixture like `full_page_html`, with one new piece: **`yield` instead of `return`.** A `yield` fixture has two halves — everything before `yield` is *setup* (create the in-memory db, build the schema), the yielded value is what the test receives, and everything after `yield` is *teardown* (close the connection), which pytest runs automatically after the test finishes, pass or fail. It's the fixture version of the open/close discipline you've been practicing since Week 3.

Now the tests. Create `tests/test_db.py`:

```python
import db


def test_insert_and_read_page(db_conn):
    db.insert_brand(db_conn, "innisfree", "Innisfree")
    db.insert_page(db_conn, "innisfree", {
        "url": "https://www.innisfree.com/",
        "status": 200,
        "title": "Innisfree",
    })
    rows = db.pages_for_brand(db_conn, "innisfree")
    assert len(rows) == 1
    assert rows[0][0] == "https://www.innisfree.com/"


def test_upsert_replaces_instead_of_duplicating(db_conn):
    db.insert_brand(db_conn, "innisfree", "Innisfree")
    db.upsert_page(db_conn, "innisfree", {"url": "https://x.com/", "title": "Old"})
    db.upsert_page(db_conn, "innisfree", {"url": "https://x.com/", "title": "New"})

    rows = db.pages_for_brand(db_conn, "innisfree")
    assert len(rows) == 1                 # one row, not two
    assert rows[0][1] == "New"            # holds the latest title


def test_missing_description_stored_as_null(db_conn):
    db.insert_brand(db_conn, "innisfree", "Innisfree")
    db.insert_page(db_conn, "innisfree", {"url": "https://y.com/", "title": "Y"})

    cursor = db_conn.execute(
        "SELECT description FROM pages WHERE url = ?", ("https://y.com/",)
    )
    assert cursor.fetchone()[0] is None
```

Read what each one pins down:

- **`test_insert_and_read_page`** — the round trip works: what you put in comes back out.
- **`test_upsert_replaces_instead_of_duplicating`** — *this is the important one.* It verifies the exact behavior you built in section 3: re-storing a URL updates rather than duplicates. If a future edit (yours or Claude Code's) ever turns `INSERT OR REPLACE` back into a plain `INSERT`, this test fails and names the regression. The behavior you cared about all day is now guarded.
- **`test_missing_description_stored_as_null`** — a missing field becomes `NULL`, not `""` (the D3 distinction), verified.

Run them:

```
pytest -v
```

Your Week 5 tests plus three new `test_db` tests, all green, still under a second — because nothing touched the disk or the network.

> Aside: SQLite doesn't enforce foreign keys by default, so inserting a page whose `brand_slug` has no matching brand won't error unless you run `PRAGMA foreign_keys = ON`. That's why the tests insert the brand first out of good hygiene, not because they'd fail otherwise. Worth knowing; not worth fixing today.
