## 4. Break-and-catch — cloud-shaped failures (~12 min)

Here's what's new this week: the failures aren't in your *code*, so `pytest` won't catch them. The site can be broken by a *deploy* or *permission* mistake while every test stays green. Vincent introduces one while you look away:

- **Re-block public access** on the bucket (flip `BlockPublicAcls` back on) → the live URL starts returning **403 Forbidden**. Tests: still green. The break is in the *bucket's permissions*, not the code.
- **Sync without building first** after a template change → the live site shows a *stale* version. Tests: green.
- **Break the bucket policy** (remove the `s3:GetObject` statement) → 403 again.
- **Point `deploy.sh` at a wrong/typo'd bucket name** → `set -u`/`set -e` should make it fail loudly rather than deploy to nowhere — does it?

**The lesson, stated plainly:** your CI gate guards your *code*, and it's excellent at that. It does **not** guard your *infrastructure* — a deploy that ships stale files, or a permission flipped the wrong way, is invisible to `pytest`. Catching these means *looking at the live site* and reading the actual error (a 403 means "authenticated maybe, but not *authorized*" — the authentication-vs-authorization split from W11D3, live). 

**Caught it?** Say what you saw (403? stale page?), diagnose which layer broke (permissions vs build vs sync), fix it, confirm the live site is back.
**Missed it?** Walk through it with Vincent: where would you *look* to notice? This is a genuinely different debugging muscle from "read the failing test" — it's "read the live system."

## 5. Claude judgment — improve the deploy (~10 min)

Open Claude Code:

```
claude
```

Ask:

> My deploy.sh syncs site/ to S3. Add a step that sets a short cache-control max-age on the uploaded files so visitors don't get stale pages for too long. Show the diff, don't apply, and explain the flag you use.

Read the diff (W4D3) and judge:

1. **Do you understand the flag it added?** Likely a `--cache-control` option on `aws s3 sync`. If you can't explain what it does to a visitor's browser, you can't accept it (W4D1) — make Claude explain `max-age` before you decide.
2. **Is it in scope and safe?** You asked for a cache header. Did it also change the bucket name, add `--acl public-read`, or alter `--delete`? A "helpful" extra flag on a command that touches your live site is exactly where to be strict.
3. **Does it actually solve a problem you have?** Caching is a real trade-off: longer caches are faster but show stale content longer; shorter caches are fresher but hit S3 more. For a brief site that updates rarely, what's the right call — and is Claude's value reasonable, or cargo-culted?

Decide accept / reject / modify, write the reason. (A defensible verdict either way — the skill is judging the trade-off for *your* site, not deferring to the suggestion.)

## Looking ahead — let the robot ship it

You have a live site and a one-command deploy. But look at who runs that command: *you*, by hand, remembering to. And nothing checks that the code is green before it goes out — `./deploy.sh` would happily publish a broken site if you ran it on broken code. The two halves you built — the **CI gate** (Week 10) and the **deploy** (Week 11) — aren't connected yet.

Week 12 connects them, and that's the finale. You'll make GitHub Actions deploy to S3 **automatically, but only when the tests pass** — a green merge to `main` ships itself; a red one ships nothing. "Only ever publish working code" stops being a promise you keep by hand and becomes a rule the machinery enforces. Then a clean-run capstone, a course-long retrospective, and you're done — Brand Lens public, tested, gated, and self-deploying.
