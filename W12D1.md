## 2. CI vs CD — naming the second half (~10 min)

Week 10 gave you **CI — continuous integration**: on every push, automatically *integrate and test* the code (the robot runs `pytest`). Today adds **CD — continuous deployment**: on every *merge to `main`*, automatically *deploy* the result. Together, "CI/CD" is the pipeline from "you propose a change" to "it's live," with as little hand-cranking as possible.

You already have all the pieces; they're just not wired together:

- The **test** half runs on the robot (Week 10).
- The **deploy** half (`deploy.sh`) runs on *your laptop*, by hand (Week 11).

Today you move the deploy half onto the robot too, triggered by a merge. The hard part isn't the workflow file — it's that the robot needs an AWS key, and giving a key to a machine has to be done *safely*.

## 3. A key for the robot — stored as a secret (~20 min)

Your laptop's AWS key lives in `~/.aws/`, which the robot can't see (it's on your machine, not in the repo — that's the whole point of W11D1). So the GitHub robot needs *its own* key. Two rules make this safe:

- **Give it a fresh, narrow key** — least privilege (W11D3): a key scoped to S3, ideally to just this one bucket, so a leak from CI loses only this site.
- **Never put the key in the repo.** GitHub provides **encrypted secrets** for exactly this — values stored *with* the repo but hidden from the code and the logs, injected into workflows at run time. The `.env` idea (W7), now hosted by GitHub.

Create the key (you can reuse your `brand-lens-deploy` user from W11D1, or make a second user `brand-lens-ci` — a separate identity for the robot is cleaner): in IAM, **Security credentials → Create access key → CLI**, and copy the access key ID and secret.

Then store them as repo secrets. On GitHub: **Settings → Secrets and variables → Actions → New repository secret.** Add two:

- `AWS_ACCESS_KEY_ID` — the access key ID.
- `AWS_SECRET_ACCESS_KEY` — the secret access key.

Once saved, GitHub will *never show them again* and will mask them in logs. They're injected into a workflow only when it runs, by name. The repo's code never contains the key — it references `secrets.AWS_ACCESS_KEY_ID`, a pointer, not the value.

## 4. Write the deploy workflow (~20 min)

A second workflow file, alongside `ci.yml`:

```
subl .github/workflows/deploy.yml
```

```yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Check out the code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Build the site
        run: python3 build_site.py

      - name: Deploy to S3
        run: aws s3 sync site/ s3://brand-lens-tyler-2026 --delete
```

Read what's new versus `ci.yml`:

- **`on: push: branches: [main]`** — this workflow runs *only* on a push to `main` (which, with branch protection, means *a merge*). Not on every branch, not on PRs — you don't want every experiment deploying. Only the trusted branch ships.
- **The "Configure AWS credentials" step** uses a pre-made action (`aws-actions/configure-aws-credentials@v4`) to do for the robot what `aws configure` did for your laptop — except it reads the key from `${{ secrets.AWS_ACCESS_KEY_ID }}`. That `${{ secrets.X }}` syntax pulls the value you stored in §3 *at run time*; the literal key never appears in the file or the logs.
- **The last two steps** are just `deploy.sh`'s body — build, then sync. (You could even call `./deploy.sh` directly; spelling the steps out keeps the bucket/region visible in the workflow.) Use *your* bucket name.

## 5. Ship it, then watch a merge deploy itself (~10 min)

Add the workflow through the usual flow (and note the recursion again — you're using a PR to add the thing that deploys on merge):

```
git switch -c add-cd
git add .github/workflows/deploy.yml
git commit -m "Add continuous deployment: sync to S3 on every merge to main"
git push -u origin add-cd
```

Open the PR. CI runs (green). Merge it. Now go to the repo's **Actions** tab and watch: the merge to `main` triggered the **Deploy** workflow. It checks out, installs, configures AWS from the secrets, builds, and syncs. A minute later — green. **Your site just deployed with nobody touching `deploy.sh`.**

Prove it's real: make a visible change (edit a heading in `templates/index.html`) on a branch, PR it, merge it, and *without running anything yourself*, refresh the live S3 URL a minute later. The change is there. The robot shipped it. That's continuous deployment.
