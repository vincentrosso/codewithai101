# Week 4, Day 4 — First Reviewed Edit
*v1.0.0*

**Goal for today:** ask Claude Code to make a real change to brand-lens — adding a `--brand` flag to `runner.py` that filters to one brand from `brands.yaml`. Read every line of the proposed diff before accepting. The point isn't the feature; it's the review reflex.

**Time budget:** ~90 min hands-on, ~15 min journal, ~15 min slack.

---

## 1. Warm-up (~5 min)

```
cd ~/dev/brand-lens
source .venv/bin/activate
python3 runner.py --help
```

Confirm the existing flags are intact (`--input`, `--output`, `--limit`, `--verbose`, `--csv`). You're about to add one more.

## 2. Write the spec yourself first (~10 min)

Before opening Claude Code, write the spec for the change in a scratch note (or just on paper):

- **What:** add a `--brand` flag to `runner.py` that takes a brand slug (e.g. `innisfree`) and filters the run to only that brand's pages.
- **Behavior when present:** only fetch the named brand's pages. Skip all others.
- **Behavior when absent:** unchanged — fetch every brand, like today.
- **Behavior when the named brand isn't in `brands.yaml`:** print an error and exit cleanly. Don't crash with a `KeyError`.
- **Out of scope:** any other change. No reformatting, no extra flags, no "while we're here" refactors.

Writing the spec yourself first matters. You're handing Claude Code a target, not a vibe. If the diff doesn't match the spec, the spec is the thing you measure against.

## 3. Hand the task to Claude Code (~10 min)

Start a Claude Code session:

```
claude
```

Paste your spec verbatim:

> Add a `--brand` flag to `runner.py` that takes a brand slug. When the flag is present, only that brand's pages are fetched. When absent, behavior is unchanged. If the named brand isn't in `brands.yaml`, print an error and exit cleanly. Out of scope: any other change — no reformatting, no extra flags, no refactoring.
>
> Show me the diff before applying anything.

Wait for the diff. Don't accept yet.

## 4. Read every line (~25 min)

The proposed diff appears in your terminal. Now apply Rule 1: review every line.

Walk through it top to bottom:

- For every `+` line, ask yourself: do I understand what this line does, and why it has to exist?
- For every `-` line: do I understand why this is being removed?
- For the hunk headers: am I looking at the part of the file I expect?

If a line is unclear, **don't try to figure it out yourself.** Ask Claude Code:

> Explain line N of the diff in plain language. Why is it there?

Do this for every unclear line. Yes, all of them. The cost of asking is low; the cost of accepting code you can't read is high.

If the diff includes a list comprehension you can't read fluently in one pass, see [sidelines/list-comprehensions.md](sidelines/list-comprehensions.md) — the *"walking from a for-loop to a comprehension"* section is the fastest way to translate one back to the long form so you can verify it.

## 5. Check the diff against your spec (~10 min)

Now compare the diff to the spec from section 2:

- Does it add `--brand` to argparse? **Yes / No / Unclear.**
- Does it filter the loop to only the named brand? **Yes / No / Unclear.**
- Does it handle the "brand not in `brands.yaml`" case cleanly? **Yes / No / Unclear.**
- Does it touch anything outside the spec? **Yes / No.**

If any answer is "Unclear" or there's scope creep, push back:

> You also changed lines X–Y. That's outside the spec I gave you. Show me a new diff that only adds the `--brand` flag and nothing else.

Iterate until the diff matches the spec.

## 6. Accept, ask for a commit message, then exit (~10 min)

When the diff is clean, accept it. While still in Claude Code, ask for a commit message:

> Suggest a one-line commit message for this change.

Note what it suggests — you'll compare against your own version in the journal. Then exit:

```
/exit
```

Test all four behaviors:

```
python3 runner.py --help                        # --brand should appear
python3 runner.py --brand innisfree --limit 1   # only innisfree pages
python3 runner.py --limit 1                     # all brands, like before
python3 runner.py --brand notarealbrand         # clean error, no traceback
```

If any of these don't behave as the spec said, the spec was wrong, the diff was wrong, or your reading of the diff was wrong. Figure out which before continuing.

## 7. Review with `git diff`, then commit (~15 min)

One last review — read the change with `git diff`, the way you practiced in W4D3:

```
git diff runner.py
```

Read the actual change, not the one Claude Code described. Sometimes they differ.

Stage and commit with a message **you write yourself** — not Claude Code's suggestion:

```
git add runner.py
git commit -m "add --brand flag to filter runner to one brand"
git push
```

A short message that names the *what*, not the *how*. Commit messages are part of your repo's history forever; they should sound like you, not like a tool.
