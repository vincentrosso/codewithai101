## 1. Before the call — a clean run from scratch (~15 min)

Today's demo is the *whole pipeline*, so prove it works from nothing. This is also the reproducibility check from W8D3/W8D4 — if a step is missing, you find out now, not in front of Vincent.

Back up nothing (it's all regenerable). Wipe the generated artifacts and rebuild:

```
source .venv/bin/activate
rm -f brand_lens.db
rm -rf briefs/

python3 runner.py            # fetch + parse + store pages
python3 summarize_all.py     # LLM brief per brand (spends a few cents)
python3 generate_briefs.py   # render briefs/<brand>.md
```

Then confirm:

```
ls briefs/
pytest -v
```

You just took Brand Lens from an empty directory to a folder of finished briefs with three commands, and the suite is green. If any step errored, fix it before the call — and notice *what* was missing (a step the README skipped? a dependency not pinned?), because that's exactly what W8D3/D4 were guarding against.

Open one brief in Sublime, ready to show.

## 2. Demo — the whole machine (~12 min)

When Vincent joins:

- **Run it live, or show the clean run you just did.** Three commands, empty → briefs. Narrate what each stage does: fetch (network), parse (HTML → fields), store (SQLite), summarize (LLM), render (template → markdown).
- **Open a finished brief.** This is the product. Read it.
- **Walk the file tree** (W8D3 §3) and name each file's one job — demonstrate you can see the project as separated concerns, not a pile of scripts.
- **Show the README** and claim: "someone could clone this and get here without me." Let Vincent poke at whether that's true.

You're demonstrating that you built a *project*, not just code that runs once on your laptop.

## 3. Grounding judgment — on the finished artifact (~10 min)

Same discipline as W7D5, now on the rendered brief a reader would actually see. Pick one brand. Put its brief next to its source data:

```
python3 -c "import db, json; conn=db.get_connection(); print(db.page_fields_for_brand(conn,'innisfree'))"
subl briefs/innisfree.md
```

Go claim by claim with Vincent: is each line in the brief **grounded** in the fetched fields, **generic** (true of any brand), or **invented**? The danger is sharper now than in Week 7, because the template makes every brief look polished and authoritative — a hallucination in a nicely-formatted document is *more* convincing, not less. Mark anything invented. That's the most important finding of the day: a beautiful brief that isn't true is the failure mode this whole tool has to guard against.

## 4. Break-and-catch (~10 min)

Vincent introduces a small change while you look away — this week the target is the template or `render.py`:

- Delete the `{% else %}` branch in `templates/brief.md` (a brand with no brief renders nothing instead of the fallback).
- Change `{{ brief.value_proposition }}` to `{{ brief.value_prop }}` (wrong key — renders blank).
- Make `render_brief` always pass `brief=None` regardless of input.
- Remove the `{% for page in pages %}` loop.

Run `pytest -v`.

**Caught it?** Read the failing test aloud, explain why, revert, confirm green.

**Missed it?** Name the gap, write the test (substring assertion against `render_brief`'s output — no mocking needed, it's pure), have Vincent re-introduce the break to watch it fail, then revert to green. The wrong-key break (`value_prop`) is a good one to reflect on: does any test assert the value-proposition *text* shows up, or only that the brand name does?

## 5. Claude judgment — extend the template (~10 min)

Open Claude Code:

```
claude
```

Ask:

> Add a "Data completeness" section to templates/brief.md that lists which of the five fields (title, description, og:title, h1, canonical) are missing for the analyzed pages, so a reader can see how much the brief had to work with. Show the diff, don't apply.

Read the diff (W4D3 rules) and judge:

1. **Did it stay in the template, or creep into `render.py` / `generate_briefs.py`?** If the feature can be done with template logic over the data you already pass, it should be — scope creep into the code is the W4D4 reflex.
2. **Is it grounded?** This feature is *itself* a grounding aid — it tells the reader how thin the data was. Does Claude's version actually surface missing fields, or just decorate?
3. **Does it need a test?** If it adds a visible behavior ("shows 'description: missing' when description is None"), that's a substring assertion waiting to be written. Would you add one before merging?

Decide accept / reject / modify, write the reason. If you accept it, this is a genuinely nice addition — it makes the grounding question from section 3 visible *in the brief itself*.

## Looking ahead — making it shareable

Brand Lens is a working tool with a tested core, a reproducible setup, and a readable output. But look at how you've been working: every change goes straight to `main`, and you run the tests by hand before committing — *if* you remember. That's fine for one person on a laptop. It stops being fine the moment the project matters or more than one person touches it.

Week 9 closes that gap. You'll learn to work in **branches** and **pull requests** instead of committing to `main` directly, and you'll set up **continuous integration** — a robot that runs your whole pytest suite automatically on every push and *blocks* a change that turns the suite red. That's the literal payoff of everything Weeks 5–8 built: the tests stop being something you remember to run and become a gate nothing broken can get through. The [why-test](sidelines/why-test.md) sideline's reason #2 — "tests are how you trust Claude Code's edits" — becomes automatic.
