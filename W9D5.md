## 1. Before the call — a clean run to a live local site (~15 min)

Today's demo is the whole machine *plus* the new site layer, so prove it from nothing — the reproducibility check (W8D3/D4), now ending at a browser instead of a folder of markdown.

```
source .venv/bin/activate
rm -f brand_lens.db
rm -rf site/ briefs/

python3 runner.py            # fetch + parse + store pages
python3 summarize_all.py     # LLM brief per brand (spends a few cents)
python3 build_site.py        # render the whole static site into site/
```

Then serve it the real way and confirm:

```
pytest -v
cd site && python3 -m http.server 8000
```

Open `http://localhost:8000`, click the index → a brand → back. `Ctrl-C`, `cd ..`. You just went from an empty directory to a browsable website with four commands. If any step errored, fix it now and note *what* was missing — that's the W8D3/D4 reproducibility discipline still doing its job.

## 2. Demo — the site (~12 min)

When Vincent joins:

- **Run the clean build, or show the one you just did.** Narrate the stages: fetch (network) → parse (HTML→fields) → store (SQLite) → summarize (LLM) → **build the static site** (templates → HTML files).
- **Open `http://localhost:8000` and click through it.** Index → a brand brief → back. This is the difference from last Friday: not a markdown file in Sublime, but a site in a browser. Point out it's being served over `http://`, the same way AWS will serve it in two weeks.
- **Say the architecture out loud:** "This is *static* — every page was built ahead of time by `build_site.py`. There's no backend; viewing a brief runs no code. That's exactly why it'll be simple to put on the internet." Showing you understand *why* it's static matters as much as the demo.

## 3. Grounding judgment — on the web page (~10 min)

Same discipline as W7D5/W8D5, now on the page a real visitor would see. Pick one brand, put its page next to its source data:

```
python3 -c "import db; conn=db.get_connection(); print(db.page_fields_for_brand(conn,'innisfree'))"
open site/innisfree.html
```

Go claim by claim: is each line **grounded** in the fetched fields, **generic** (true of any brand), or **invented**? The caution sharpens again — a styled web page is *even more* convincing than a markdown file, and an authoritative-looking layout makes a hallucination read as fact. Mark anything invented. A handsome lie is still a lie; the prettier the page, the more it earns scrutiny.

## 4. Break-and-catch (~10 min)

Vincent introduces a small change while you look away — this week the target is a template, the stylesheet, or `build_site.py`:

- Change `href="{{ brand.slug }}.html"` to `href="{{ brand.name }}.html"` in the index (links now point at display names — every link 404s).
- Delete the `{% else %}` in `templates/brief.html` (a no-brief brand renders an empty Summary).
- Remove the `shutil.copy(...)` line from `build_site.py` (site builds, but every page loses its styling).
- Break the slug-to-filename glue so `write_page` and the index disagree.

Run `pytest -v`, then rebuild and look at `localhost:8000`.

**Two kinds of break, two kinds of catch.** Some of these the *test suite* catches (the `href` and `{% else %}` ones — `test_index_links_each_brand`, the fallback test). Others — like the missing stylesheet copy — the suite *won't* catch, because the pages still render correct HTML; only your *eyes* on the served site catch it. That's a real and useful lesson: tests guard behavior they were written for; some failures are only visible in the running thing. Note which breaks needed pytest and which needed a browser.

**Caught it?** Read the failing test (or describe what you saw), explain, revert, confirm green and styled.
**Missed it?** Name the gap; if it's test-catchable, write the test (a substring assertion on the rendered string — pure, no mocking) and have Vincent re-break it to watch it fail.

## 5. Claude judgment — improve the site (~10 min)

Open Claude Code:

```
claude
```

Ask:

> In templates/brief.html, add a small colored badge next to the Confidence value — green for "high", amber for "medium", red for "low" — using a CSS class, with the styles in static/style.css. Show the diff, don't apply.

Read the diff (W4D3 rules) and judge:

1. **Did it keep the concerns separated?** The *class* belongs in the template, the *colors* in `style.css`. If Claude inlined a `style="color:red"` attribute into the HTML instead of using the stylesheet, that's the HTML/CSS separation (W9D2) violated — the kind of thing that looks fine and rots later.
2. **Did it stay in scope?** You asked for a badge. Did it also "helpfully" restructure the template or restyle unrelated things? Scope creep is the W4D4 reflex.
3. **Does it need a test?** If the badge shows the confidence word in a class, is there a substring assertion worth adding? Would you add it before merging?

Decide accept / reject / modify and write the reason. (If accepted, rebuild and look — a small touch of color that ties the page's most important judgment to a glanceable signal.)

## Looking ahead — how you've been working

Stop and look not at *what* you built this week but at *how*. Every commit this week — every commit all course — went straight to `main`. You run `pytest` by hand before pushing, *when you remember*. And if you'd pushed one of this week's broken templates, nothing would have stopped you; the break would just be live.

That's been fine for one person on a laptop. It stops being fine the instant the project matters — and next week it starts to matter, because Week 11 puts this site on the real internet, and you only want *working* versions going out.

Week 10 fixes the *how*. You'll stop committing to `main` directly and start working in **branches** and **pull requests** — proposing a change, reviewing it, then merging. And you'll set up **continuous integration**: a robot that runs your entire pytest suite automatically on every push and **blocks** a change that turns the suite red. This is the literal cash-in of everything Weeks 5–9 built — the [why-test](sidelines/why-test.md) sideline's reason #2, "tests are how you trust a change," made automatic. (Your own messy commit history this course — the duplicate pushes, the half-finished `main` — is Exhibit A for why this workflow exists. Bring it; it's the motivation.)
