## 1. Warm-up (~5 min)

```
source .venv/bin/activate
git switch main && git pull
pytest -v
```

Green, synced. Today you run the whole machine end to end, the way you'll run it for Vincent tomorrow.

## 2. The clean pipeline run — empty to built (~20 min)

Wipe the regenerable artifacts and rebuild from nothing. This is the reproducibility check (W8D3 onward), final form:

```
rm -f brand_lens.db
rm -rf site/ briefs/

python3 runner.py            # fetch + parse + store pages
python3 summarize_all.py     # LLM brief per brand (a few cents)
python3 build_site.py        # render the whole static site into site/
pytest -v                    # whole suite green
```

Then preview locally before anything goes public:

```
cd site && python3 -m http.server 8000      # open http://localhost:8000, click through
# Ctrl-C, cd ..
```

If any step errored, fix it *now* and note what was missing — a step the README skipped, a dependency not pinned. Finding it today is the entire reason for a rehearsal.

## 3. Ship a real change through the full loop (~20 min)

Don't just `./deploy.sh` — exercise the *whole* professional loop you built, because that's what you're demoing. Make a small, real improvement and let the machinery carry it to production:

```
git switch -c capstone-polish
```

Make a genuine touch-up — tighten the index's intro line, or add a "last generated" note, something you'd actually want live. Build and eyeball it:

```
python3 build_site.py && open site/index.html
```

Commit, push, open the PR:

```
git add -A
git commit -m "Polish the index copy for the capstone demo"
git push -u origin capstone-polish
```

On the PR: watch the `test` job go green. Merge. Then — **hands off the keyboard** — go to the **Actions** tab and watch the `deploy` job run *because* `test` passed, build the site, and sync it to S3. A minute later, refresh the live S3 URL: your polish is live, shipped by the robot, gated by the tests. *That* is the system working as one machine. Clean up:

```
git switch main && git pull
git branch -d capstone-polish
```

## 4. Finalize the README (~20 min)

Tomorrow you'll claim "a stranger could clone this and get to a live site." Make that true. Open the project `README.md` (the one from W8D4) and bring it current — it predates Weeks 9–12 and is now missing half the story. It should cover:

- **What it is** — one or two sentences (unchanged from W8, still good).
- **Setup** — clone, venv, `pip install -r requirements.txt`, `.env` for the Anthropic key, *and now* AWS: `aws configure` with a least-privilege user (point at W11, don't re-teach it).
- **Build & run locally** — `runner.py` → `summarize_all.py` → `build_site.py`, then `python -m http.server` to preview.
- **Deploy** — `./deploy.sh` for a manual deploy; and note that a merge to `main` deploys automatically via GitHub Actions (the CI/CD loop).
- **Layout** — the file tree, now including `build_site.py`, `templates/*.html`, `static/style.css`, `deploy.sh`, `.github/workflows/`.

The test (W8D3 §5) is unchanged: could someone clone this and reach a live site *without asking you*? Walk your own README and find the gap — there's usually one. (Pin `requirements.txt` too if Weeks 9–12 added anything new to import — though for a static site, you likely added no new runtime dependencies, which is worth noticing: `jinja2` already covered the HTML, and `aws`/`git` aren't Python imports. Confirm and move on.)

Ship the README update through the loop (it's a change to `main` like any other):

```
git switch -c update-readme
git add README.md requirements.txt
git commit -m "Update README for the static site, AWS deploy, and CI/CD"
git push -u origin update-readme
```

PR, green, merge, `git pull`. (And note: merging the README triggers a deploy — harmless, the site just rebuilds identically. A good reminder that *every* merge to `main` ships now.)
