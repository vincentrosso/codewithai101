## 2. Why a template, not f-strings (~10 min)

You could build the document with the string tools you already know — `f"# {brand_name}\n\n{value_prop}..."`. For one line, fine. For a whole report with headings, a list of pages, and "show this section only if there's a brief," string-building turns into an unreadable tangle of `\n` and nested `if`s, with the *layout* of the document buried inside Python logic.

A **template** separates the two concerns. The template is a file that looks like the finished document, with **placeholders** where data goes. Your Python code's only job is to hand the template a bag of data; the template decides how it looks. Change the report's wording or add a section? Edit the template, not the code. This is the same instinct as splitting `fetch.py` from `parse.py` in Week 3 — *keep distinct concerns in distinct places* — applied to "the data" vs "how it's presented." (D3 makes this idea general.)

`jinja2` is the standard Python templating library. (It's what powers the HTML in Flask apps, and the autoarb scraper uses the same idea to build its daily report.)

## 4. Write your first template (~20 min)

Templates live in their own directory by convention:

```
mkdir templates
touch templates/brief.md
subl templates/brief.md
```

```jinja
# {{ brand_name }} — US Presence Brief

*Generated {{ generated_at }}*

## Summary

{% if brief %}
**Value proposition:** {{ brief.value_proposition }}

**Messaging consistency:** {{ brief.messaging_consistency }}

**US-presence signal:** {{ brief.us_presence_signal }}

**Confidence:** {{ brief.confidence }}
{% else %}
_No brief has been generated for this brand yet._
{% endif %}

## Pages analyzed

{% for page in pages %}
- [{{ page.title or page.url }}]({{ page.url }})
{% endfor %}
```

This is markdown with `jinja2` markings mixed in. Two kinds:

- **`{{ ... }}` — output.** "Put the value here." `{{ brand_name }}` drops in whatever `brand_name` you pass. `{{ brief.value_proposition }}` reaches into the `brief` dict for its `value_proposition` key — note the dot, `jinja2`'s shorthand for `brief["value_proposition"]`.
- **`{% ... %}` — logic.** `{% if brief %}...{% else %}...{% endif %}` shows one block or the other depending on whether `brief` is truthy. `{% for page in pages %}...{% endfor %}` repeats its block once per page — the template's version of the `for` loops you've written since Week 1.

Notice everything *outside* the markings is just text that comes through unchanged. The template is the document, with holes.

## 5. Render it from Python (~20 min)

Create the rendering module:

```
touch render.py
subl render.py
```

```python
from jinja2 import Environment, FileSystemLoader

env = Environment(loader=FileSystemLoader("templates"))


def render_brief(context):
    """Turn a context dict into the rendered markdown string."""
    template = env.get_template("brief.md")
    return template.render(**context)
```

Read it:

- **`Environment(loader=FileSystemLoader("templates"))`** — sets up `jinja2` to find templates in the `templates/` directory. You create this once.
- **`env.get_template("brief.md")`** — loads the file you just wrote.
- **`template.render(**context)`** — fills the template with your data and returns the finished string. The `**context` unpacks a dict into named arguments — so `render(**{"brand_name": "X", ...})` becomes `render(brand_name="X", ...)`, and inside the template `{{ brand_name }}` finds it. (You met `**` in the [cli-args-and-kwargs](sidelines/cli-args-and-kwargs.md) sideline.)

One thing worth noticing now, because D4 leans on it hard: **`render_brief` is a *pure* function** (W5D3). Hand it the same context dict, it returns the same string every time — no file written, no database touched, no network. That's deliberate, and it's why it'll be trivial to test.

Try it with hand-made data (throwaway script):

```
touch try_render.py
subl try_render.py
```


## 6. Write it to a file (~10 min)

Printing is for checking. The MVP writes files. Add to `render.py`:

```python
import os


def write_brief(slug, markdown):
    os.makedirs("briefs", exist_ok=True)
    path = os.path.join("briefs", f"{slug}.md")
    with open(path, "w", encoding="utf-8") as f:
        f.write(markdown)
    return path
```

- **`os.makedirs("briefs", exist_ok=True)`** — create the `briefs/` directory; `exist_ok=True` means "don't error if it's already there" (the same spirit as `CREATE TABLE IF NOT EXISTS`).
- **`open(..., "w", encoding="utf-8")`** — write mode, UTF-8 (W3D3 — Korean brand names need it).

Update `try_render.py` to write instead of print:

```python
from render import render_brief, write_brief

# ... same context ...
markdown = render_brief(context)
path = write_brief("innisfree", markdown)
print(f"wrote {path}")
```

```
python3 try_render.py
```

Open the result:

```
subl briefs/innisfree.md
```

A real document. Notice the division of labor: `render_brief` (pure — builds the string) and `write_brief` (side effect — touches the disk) are *separate functions*. That separation is exactly the testability principle from W5D3, and you'll cash it in on D4.

**Keep generated briefs out of git.** They're regenerable output, like `brand_lens.db`. Add to `.gitignore`:

```
briefs/
```

(If you later want to show off one brief as a sample, you can commit a single file deliberately — but the directory of generated output doesn't belong in version control.)
