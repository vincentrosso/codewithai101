## 2. The plan (~5 min)

Yesterday you rendered one brand from a dict you typed by hand. Today the dict comes from the database. For each brand you need to build the same **context** shape `render_brief` expects:

```python
{
    "brand_name": ...,     # from brands.name
    "generated_at": ...,   # today's date
    "brief": ...,          # the stored JSON, parsed back into a dict — or None
    "pages": ...,          # from db.page_fields_for_brand
}
```

So today is two functions: one that *builds a context from the database* for a given brand, and one that *loops over all brands* and renders each. The rendering itself (`render_brief`) doesn't change — that's the payoff of keeping it pure.

## 3. Build a context from the database (~20 min)

Create the entry script:

```
touch generate_briefs.py
subl generate_briefs.py
```

```python
import json
from datetime import date

import db
from render import render_brief, write_brief


def build_context(conn, slug, name):
    row = conn.execute(
        "SELECT brief FROM brands WHERE slug = ?", (slug,)
    ).fetchone()

    brief_json = row[0] if row else None
    brief = json.loads(brief_json) if brief_json else None

    pages = db.page_fields_for_brand(conn, slug)

    return {
        "brand_name": name,
        "generated_at": date.today().isoformat(),
        "brief": brief,
        "pages": pages,
    }
```

The line that matters most:

```python
brief = json.loads(brief_json) if brief_json else None
```

This is the **other half of a round trip**. On W7D4 you stored the brief with `json.dumps(summary)` — turning the dict into a string for the database. Now you do the reverse: `json.loads(brief_json)` turns that string back into a dict. The `if brief_json else None` guard handles the brand with no brief — `brief_json` is `None` (the column was `NULL`), so you skip the parse and pass `None` straight through. That `None` is exactly what your template's `{% if brief %}` checks.

Read the rest:

- **`.fetchone()`** returns one row (a tuple) or `None` if no brand matched. `row[0]` is the `brief` column.
- **`date.today().isoformat()`** — today as a string like `2026-05-27`. (`from datetime import date`.)
- **`db.page_fields_for_brand`** — the W7D2 query, reused unchanged.

## 4. Loop over all brands (~15 min)

Add to `generate_briefs.py`:

```python
def generate_all(conn):
    paths = []
    for slug, name in conn.execute("SELECT slug, name FROM brands").fetchall():
        context = build_context(conn, slug, name)
        markdown = render_brief(context)
        path = write_brief(slug, markdown)
        paths.append(path)
    return paths


if __name__ == "__main__":
    conn = db.get_connection()
    written = generate_all(conn)
    conn.close()
    for path in written:
        print(f"wrote {path}")
```

The `if __name__ == "__main__":` guard is the one from the [main-guard](sidelines/main-guard.md) sideline — it lets `generate_briefs.py` be both a script you run *and* a module D4's tests can import without it running the whole generation. (This was one of the lines you flagged as not-fully-understood back in W4D5. It's load-bearing again here — worth re-reading that sideline if it's still fuzzy.)

That graceful handling of missing data is the same discipline as `parse_page` returning `None` for a missing title (Week 2) and the brief's `(summary, error)` contract (W7D4): **the program produces something sensible even when a piece is absent.**

## 6. A grounding reminder (~5 min)

You're about to be able to generate a folder full of professional-looking briefs in one command. That's exactly when the W7D5 lesson matters most: **a brief that reads well is not the same as a brief that's true.** The template makes *every* brief look authoritative — including a brand whose brief the model partly invented. Before any of these goes in front of a real person, the brief still has to be read against its source data. The MVP makes the output pretty; it does not make it correct. Hold that distinction; D5 comes back to it.
