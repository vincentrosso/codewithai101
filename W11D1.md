## About this week — Brand Lens on the real internet

This is the week Brand Lens stops being a thing on your laptop and becomes a thing on the internet. You'll deploy the static site to **Amazon Web Services (AWS)** — specifically to **S3**, AWS's file-storage service, which can serve a static site straight to the public.

Two reasons this is the right target, not a toy:

- **It's the simplest possible deploy**, *because* your site is static (W9D3). No server to run, no backend to keep alive — just files in a bucket, handed out on request. The entire deploy is "copy `site/` up there."
- **It's the stack you'll actually use.** AWS is where NIT ships. What you learn this week — accounts, users, keys, regions, the CLI — is the real vocabulary of the job, not a classroom simplification.

This week:

- **Today (D1):** AWS account, a locked-down user, the AWS CLI, prove the connection.
- **D2:** create a bucket, copy the site up, turn on website hosting — *live on the internet*.
- **D3:** foundations — no code, no AI — what "the cloud," a user, a region, and a bucket actually are.
- **D4:** one command to deploy — `deploy.sh` — plus cost and tear-down discipline.
- **D5:** the Friday demo of the live site, plus the Week 11 retrospective.

A note before you start: AWS is a real account with real (if tiny) costs and real security stakes. Read each step; don't paste commands you can't explain (the W4D1 rule matters more than ever when the command touches a billing account). When in doubt, stop and ask.

## 2. Create the AWS account (~15 min)

If you (or NIT) already have an AWS account to use, skip to §3 — but still create a *dedicated user* there in §4 rather than using anyone's root login.

Otherwise, go to **aws.amazon.com** → **Create an AWS Account**. It asks for an email, a password, and — yes — a credit card. AWS bills by usage; what you'll do this week (a few megabytes in S3, a handful of requests) costs effectively nothing, often literally pennies, and S3's free tier may cover it entirely. You'll set up tear-down discipline on D4 so nothing lingers. Finish the sign-up and the email/phone verification.

The account you just made has a **root user** — the email you signed up with. Root can do *anything*, including close the account and run up unlimited bills. Which is exactly why you're about to stop using it.

## 3. Why not just use root (~10 min)

The root user is the master key to the whole account. Using it for daily work is like carrying the master key to your entire building to unlock your own office — if you drop it, you don't lose your office, you lose *everything*.

This is the same instinct you've practiced all course, scaled up:

- In W7 you kept your `ANTHROPIC_API_KEY` in `.env`, never in code, because a leaked key spends your money. An AWS key is the same kind of secret, with higher stakes.
- The principle is **least privilege**: a credential should be able to do *only* what the task needs, nothing more. If your laptop only needs to upload files to one bucket, its key should be able to do *only* that — so if it leaks, the blast radius is one bucket, not your whole account.

So instead of handing your laptop the master key, you'll create a **user** with a narrow set of permissions and give *that* user a key. (D3 makes "user" and "permissions" precise; today you just create one.)

## 4. Create a locked-down user (~20 min)

AWS's service for users and permissions is **IAM** (Identity and Access Management). Sign in as root *this once* to create your working user, then put root away.

In the AWS Console, go to **IAM → Users → Create user**:

- **User name:** something like `brand-lens-deploy`.
- On permissions, choose **Attach policies directly** and attach **AmazonS3FullAccess**. A **policy** is a list of allowed actions; this one grants S3 actions only — not billing, not other services. (Tighter is possible — scoping to a single bucket — but S3-only is a reasonable, much-narrower-than-root starting point. The point today is *not root*.)
- Create the user.

Now create an **access key** for it — the credential your laptop's CLI will use. On the user's page: **Security credentials → Create access key → Command Line Interface (CLI)**. AWS shows you two strings:

- **Access key ID** — like a username (identifies the key).
- **Secret access key** — like a password (proves it's yours). **AWS shows this once.** Copy both somewhere safe for the next step.

Treat the secret access key exactly like your Anthropic key: it's a password that spends money. It goes nowhere near git.

## 5. Install the AWS CLI (~10 min)

The **AWS CLI** is a command-line tool that talks to AWS from your terminal — the same idea as `git` talking to GitHub. Install it with Homebrew (W1D2):

```
brew install awscli
```

Confirm it's there:

```
aws --version
```

You should see a version string. If `aws` isn't found, your `PATH` may need a new terminal window — close and reopen, then retry.

## 6. Configure the CLI — where the key lives (~10 min)

Hand the CLI your access key:

```
aws configure
```

It prompts for four things:

- **AWS Access Key ID** — paste the access key ID from §4.
- **AWS Secret Access Key** — paste the secret access key.
- **Default region name** — type `us-east-1` (or NIT's region if you know it). A **region** is *which of AWS's data centers* your stuff lives in (D3). Pick one and be consistent.
- **Default output format** — type `json`.

Where did those keys go? **Not into your project.** The CLI saves them to `~/.aws/credentials` — a file in your *home* directory, outside the brand-lens repo entirely. So they can't be committed by accident; they're not in the repo's reach at all. (This is even safer than `.env`, which at least lives in the project folder. Same principle, one notch tighter.) Never copy these keys into the repo to "make it convenient."

## 7. Prove the connection (~5 min)

Ask AWS who it thinks you are:

```
aws sts get-caller-identity
```

You should get back a small JSON blob with an `Arn` ending in `user/brand-lens-deploy` — AWS confirming your laptop is authenticated as the locked-down user you made, not root. That's the whole win for today: a *secure, working* line from your terminal to AWS. Tomorrow you use it to make a bucket.

If it errors with "Unable to locate credentials" or "InvalidClientTokenId," re-run `aws configure` and re-check the two keys (a trailing space on a paste is the classic culprit).
