# Week 5, Day 5 — Friday Review: Tests as Armor
*v1.0.0*

**Goal for today:** the normal Friday mentor session is back. You demo the test suite, Vincent introduces a small break into one of your functions, your tests should catch it. If they don't, you write the test that would have. Plus one Claude-judging exercise: ask Claude Code to write a test, and judge whether it's actually useful.

**Time budget:** ~30 min mentor call, ~45 min hands-on, ~20 min retrospective journal, ~15 min slack.

---

## 1. Before the call (~10 min)

Run the full suite and confirm everything is green:

```
source .venv/bin/activate
pytest -v
```

You should see nine (or ten, if you did the optional 404 test) tests passing.

Open two terminal panes:

- One running your test command, ready to re-run on demand.
- One showing `parse.py` and `fetch.py` — Vincent will be editing these.

Have `tests/test_parse.py`, `tests/test_fetch.py`, and `tests/conftest.py` open in Sublime, ready to walk through.

## 2. Demo — walk Vincent through the suite (~10 min)

Open Zoom (or whatever you use). When Vincent joins:

- Show the test files top to bottom. Read the test names aloud. Your function names are the spec — let them be the demo.
- Pick one fixture (`full_page_html`) and explain how pytest wires it to a test by parameter name.
- Pick one mock (the `fake_get` in `test_fetch_one_returns_error_when_request_raises`) and walk through what's being faked, what gets substituted, and what the test proves.

You're aiming to explain *why* the suite is shaped the way it is, not to recite test names. The point is to demonstrate you understand the tools, not memorized them.

## 3. The break-and-catch exercise (~15 min)

This is the meat of the review.

Vincent will pick one of `parse.py` or `fetch.py` and introduce a small change while you look away — something like:

- Make `parse_page` return `record["title"]` *before* stripping whitespace.
- Make `parse_page` ignore the `og_title` field entirely.
- Make `fetch_one` swallow exceptions silently and return `(None, None, None)`.
- Change the canonical-URL extraction to read `rel="canonical-url"` instead of `rel="canonical"`.

Each of these is a small, plausible change — exactly the kind Claude Code might propose two months from now in a 60-line diff.

You then run:

```
pytest -v
```

Two outcomes:

**Outcome A — a test caught it.** Read the failure message aloud. Walk through which test failed and *why* it failed. Then revert the change (or have Vincent revert it) and re-run to confirm green.

**Outcome B — no test caught it.** This is the more important case. The suite has a gap. Together:

1. Name the gap: which specific behavior wasn't being verified?
2. Write the test that would have caught it. Use a fixture if it applies, mock `requests.get` if it's a `fetch` test.
3. Vincent re-introduces the bug. Re-run pytest. The new test should now fail — proving it actually checks what you think it checks.
4. Revert the bug. Re-run pytest. The new test should pass.

This is the green-red-green cycle from D1, but with the test driven by a real gap instead of a hypothetical one. By the end of the exercise you should have at least one new test in your suite.

## 4. Claude judgment — let Claude Code write a test (~15 min)

Open Claude Code:

```
claude
```

Ask:

> Write a pytest test for parse_page in parse.py that verifies the canonical URL is extracted from a <link rel="canonical"> tag. Show me the test as a diff but don't apply it yet.

Read the diff carefully (W4D3 rules apply). Then judge:

1. **Does it test behavior or implementation?** A behavior test asserts on the *output* (`record["canonical"] == "..."`). An implementation test asserts on *how* the function works internally (e.g., "BeautifulSoup's `find` method was called with these arguments"). Behavior tests survive refactors. Implementation tests break the moment you rewrite the function in a different way, even if the function still does the same thing.

2. **Does it duplicate a test you already have?** You wrote `test_parse_page_extracts_canonical` on D2. If Claude Code's test is identical, that's not useful. If it covers a *different* case (e.g., no canonical tag present, or multiple canonical tags), it might be.

3. **Is the HTML in the test minimal?** A good test uses the smallest HTML that exercises the behavior. A 50-line HTML doc with a single canonical tag buried in it is noise. A 5-line HTML with just the canonical tag is signal.

4. **What's the test's name?** Is it descriptive enough to read on its own? `test_canonical` is weak; `test_parse_page_extracts_canonical_url_from_link_tag` is strong.

Decide: accept, reject, or modify. Write your decision and the reason in your journal. The point of the exercise isn't to merge the test — it's to practice judging Claude's output against the standards you built this week.
