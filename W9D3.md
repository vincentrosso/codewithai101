## 2. The browser is the renderer (~15 min)

Yesterday's surprise (or it should have been): `print(html)` showed raw tags, but `open site/innisfree.html` showed a drawn page. Same string, two completely different results. Why?

Because **the browser is a program whose entire job is to turn HTML text into a drawn page.** It reads the tags, builds a model of the page in memory, applies the CSS, and paints pixels. `print` has no such job — it dumps characters to a terminal. The tags aren't magic; they're *instructions to a specific reader*, and only the browser is that reader.

This is the same lesson as the relational model (W6) or the assembly sideline: a layer of text means nothing until something *interprets* it. Python source is inert until the interpreter runs it. SQL is inert until the database engine runs it. HTML is inert until the browser draws it. The HTML you wrote is a *description of a page*; the browser is what makes the description real.

Name the three languages the browser reads, because you've now touched all three:

- **HTML** — the *content and structure*. What's a heading, what's a paragraph, what's a link. (You wrote this D1–D2.)
- **CSS** — the *appearance*. Fonts, colors, spacing. (You wrote this D2.)
- **JavaScript** — *behavior*. Code that runs in the browser to make a page interactive. **You haven't written any, and Brand Lens needs none** — that's exactly what makes your site *static* (next section). Worth knowing it exists; worth knowing you're deliberately not using it.

## 3. The word "static" — and what it rules out (~20 min)

This is the most important idea of the week, so go slowly.

A **static** site is one where the files are *already finished* before anyone visits. Your build script runs, produces `index.html` and the brand pages, and those files just sit there. When someone visits, they get the file exactly as it was built. The page is the same for every visitor, until you rebuild it. Viewing the site causes no work, no computation, no decisions — just a file being handed over.

Contrast that with a **dynamic** site, where a program runs *at the moment you visit* and builds the page on the spot — different for each visitor, or each second. When you log into a bank and see *your* balance, a program looked up your account and generated that page right then. That program — the thing that runs per-request — is called a **backend** (or server-side application).

Now the question that decides your whole deploy: **does Brand Lens need a backend?**

Walk through what it does. It scrapes pages, stores them, asks the LLM for a brief, and renders HTML. *All of that happens when you run the pipeline* — ahead of time, on your machine. By the time anyone looks at a brief, the brief is a finished file. Nothing about *viewing* it requires a decision, a lookup, or a computation. There's no per-visitor anything: every visitor sees the same brief until you re-run the pipeline.

So: **no.** Brand Lens has no backend, and that's not a limitation you're settling for — it's the *correct* shape for this tool. A batch pipeline that produces documents *is* a static site. (A web server sitting there to serve pre-built files that never change between runs would be a doorman employed to hand out the same photograph to everyone. Real work for no reason.)

Why this matters so much: **static sites are radically simpler to put on the internet than dynamic ones.** No program to keep running, nothing to crash at 3 a.m., nothing to attack, no server to pay for by the hour. You just need somewhere to *hold the files* and hand them out. In Week 11 that somewhere is an AWS S3 bucket, and the entire deploy is "copy my `site/` folder up there." The simplicity you're banking is bought right here, by the site being static.

*(The day Brand Lens needs to do something the instant you click — a live search box, a "re-scrape this brand now" button, a login — it would grow a backend, and that's a genuinely bigger machine. That's a future course, not this one. For now, notice how much you get to skip by not needing it.)*

## 4. `file://` vs a real server — and why `python -m http.server` exists (~15 min)

When you ran `open site/innisfree.html`, look at the browser's address bar. It says something like `file:///Users/tyler/.../site/innisfree.html`. That `file://` means the browser read the file **straight off your disk** — no server involved at all. For a single page, fine.

But a real website isn't reached by a file path; it's reached over the network from a **server** — a program that listens for "give me this page" requests and responds with the file. The address starts with `http://`, not `file://`. The difference isn't cosmetic: some things (the way links resolve, certain browser security rules) behave differently under `file://` than under `http://`, so a site that looks fine opened directly can misbehave once it's actually served — and you'd rather find that out on your laptop than after deploying.

Python ships with a tiny server built in exactly so you can preview your site the *real* way, locally:

```
cd site
python3 -m http.server 8000
```

That command turns the current folder into a little website served at `http://localhost:8000`. `localhost` means "this very machine"; `8000` is the **port** — think of an address as a building and the port as which door (more in Week 11's foundations). Open `http://localhost:8000` in a browser and you're viewing your site over `http://`, served by a real (if minimal) server, the same way the world will once it's on S3. Stop it with `Ctrl-C`.

You'll use this tomorrow to preview the finished build. The point to carry: **`open` (a `file://` path) is a peek; `python -m http.server` (an `http://` address) is a rehearsal for how it'll really be served.**

## 5. Absolute vs relative links — the trap (~10 min)

Your links so far are *relative*: `href="index.html"`, `href="innisfree.html"`. Relative means "find this **next to the current page**." There's another kind, **absolute**, which names a full path or URL: `href="/Users/tyler/brand-lens/site/index.html"` or `href="https://example.com/index.html"`.

Why relative is the right call for a static site you're about to move: a relative link makes no assumption about *where the site lives*. The pages just need to be siblings in one folder. So the exact same files work three places unchanged — opened from `site/` on your disk, served from `localhost:8000`, and (Week 11) served from an S3 bucket on the internet. The moment you hard-code an absolute path like `/Users/tyler/...`, the link breaks everywhere except your laptop. **Relative links are what make the site portable** — build once, run anywhere, which is the whole game in Weeks 11–12.
