## 3. Grounded or hallucinated? (~15 min) — the new judgment

This is the part with no analog in earlier weeks. Put the brief and the raw scraped fields for your chosen brand side by side. For each claim in the brief, ask:

> **Is this supported by the data I actually sent, or did the model make it up?**

Go field by field with Vincent:

- The `value_proposition` claim — can you point to the `description` or `h1` that backs it? Or did Claude invent a polished pitch the page never made?
- The `messaging_consistency` claim — does it actually reflect the `title`/`og_title`/`h1` you sent, or is it generic filler that would be true of any brand?
- The `us_presence_signal` claim — does it follow from the `canonical` URLs, or is it a guess?
- The `confidence` value — when a field was `None`, did the brief lower its confidence and say so, or did it confidently fill the gap with invention (a hallucination, W7D3 §5)?

Mark each claim **grounded**, **generic**, or **invented**. This is the skill that makes an LLM feature trustworthy: the model will *always* produce fluent, confident text — your job is to check it against the source. A brief that reads beautifully but isn't grounded is worse than no brief, because it *looks* authoritative.

If you find an invented claim, that's the most valuable result of the day. Discuss with Vincent: was it the prompt's fault (too vague, didn't demand grounding), the data's fault (too little to go on), or the model just confabulating? The answer shapes how you'd fix the prompt — likely Week 8 work.

## 4. Break-and-catch (~10 min)

Vincent introduces a small change to `brief.py` while you look away:

- Remove the `try/except` around `json.loads`, so a bad reply raises instead of returning an error.
- Change `summarize_brand` to `return summary` (one value) instead of `(summary, error)`.
- Make the JSON-parse `except` swallow the error and `return {}, None` (pretends success on garbage).

Run `pytest -v`.

**Caught it?** Read the failing test aloud, explain why it fired, revert, confirm green.

**Missed it?** Name the gap, write the test that catches it (using the `FakeMessage`/`monkeypatch` pattern), have Vincent re-introduce the bug to watch the new test fail, then revert to green. The `return summary` break is a good one: does any test actually unpack *two* values and assert on the error, or would they all still pass against a one-value return?

## 5. Claude judgment — let Claude Code improve the prompt (~10 min)

Open Claude Code:

```
claude
```

Ask:

> Look at build_prompt and SYSTEM in brief.py. Suggest a change that makes Claude less likely to invent a value proposition when the meta description and h1 are both missing. Show it as a diff, don't apply it.

Read the diff (W4D3 rules). Then judge — this is subtler than judging code, because there's no test that says "right":

1. **Does the change actually push toward grounding?** A good change tells the model to say so (or lower confidence) when the data is thin. A weak change just adds flattering words.
2. **Could you *verify* the change helped?** You can't unit-test prose quality. But you *could* run the brief on a brand with missing fields before and after and compare the `confidence` and whether it invented a pitch. Would you trust the change without doing that?
3. **Did it touch only the prompt, or did it creep into the parsing/return logic** you carefully built and tested? Scope creep in a diff is the W4D4 reflex.

Decide accept / reject / modify and write the reason. The point: prompt quality is judged by *grounding*, and grounding is judged by *you reading the output against the data* — not by the suite going green.

## Looking ahead — the MVP is in sight

Step back and see what exists at the end of Week 7. Brand Lens fetches a brand's US pages, parses the fields that matter, stores them in a queryable database, and turns them into a structured, stored judgment about the brand's US presence — with the LLM call wrapped so it can't crash the run and tested so it can't silently break. That's the machine the course set out to build in Week 1.

What's missing is the *last mile*: turning a stored brief into something a person reads. That's the natural next week — assembling fetched data and briefs into a per-brand markdown document (a `briefs/` directory, one file per brand), the readable artifact that makes Brand Lens a tool you'd actually demo. The grounding question you practiced today becomes central there: a brief you'd put in front of a brand has to be *true*, not just fluent.
