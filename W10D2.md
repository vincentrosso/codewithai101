# Week 10, Day 2 — The Pull Request
*v1.0.0*

**Goal for today:** turn yesterday's branch into a **pull request** — a formal proposal to merge your change into `main` — review your own diff in it, merge it on GitHub, and bring the updated `main` back to your laptop. By end of day the footer is on `main`, the branch is merged and deleted, and you've done one full loop of the workflow real teams use all day.

**Time budget:** ~70 min hands-on, ~15 min journal, ~15 min slack.

---

## 1. Warm-up (~5 min)

```
source .venv/bin/activate
git switch add-site-footer
git status
pytest -v
```

On your branch, clean, green. Yesterday you pushed `add-site-footer` to GitHub but stopped short of merging. Today you finish the loop.

## 2. What a pull request is (~10 min)

A **pull request** (PR) is a request to *pull* the commits from your branch into another branch — here, into `main`. It's the proposal step the straight-to-`main` workflow never had. A PR is two things at once:

- **Technically**, it's "please merge `add-site-footer` into `main`." When you accept it, GitHub merges the branch.
- **Socially**, it's a *review checkpoint* — a page where the change is laid out as a diff for someone (a teammate, or future-you) to read, comment on, and approve *before* it becomes part of `main`. On a team, someone else reviews your PR; solo, *you* review it, with fresh eyes.

This is the exact discipline from Week 4 — "review every diff before you accept it" (W4D1, W4D3) — except now the diff being reviewed is *your own change*, and the review happens in a structured place instead of in your head. The PR is where "look before it's permanent" lives.

## 3. Open the PR (~15 min)

On GitHub, open your repo. Either click the **"Compare & pull request"** banner from yesterday, or go to the **Pull requests** tab → **New pull request** and choose `main` ← `add-site-footer`.

You'll get a form:

- **Base** should be `main`, **compare** should be `add-site-footer` — read this as "merge *compare* into *base*." Get this backwards and you'd propose merging `main` into your branch; check it.
- **Title** — a short summary. GitHub pre-fills it from your commit message. "Add a descriptive footer to the site index" is fine.
- **Description** — a sentence on *why*. "Gives the index page a one-line explanation of what Brand Lens is, for visitors landing cold." A PR description answers "why should this be merged?", which the diff alone doesn't.

Click **Create pull request**. You now have a PR — a permanent, reviewable page for this change.

## 4. Review your own diff (~15 min)

Before merging, **read the diff** — the PR's "Files changed" tab. This is the W4D3 skill, applied to your own work:

- **Is the change only what you intended?** You meant to edit the footer in `templates/index.html`. Does the diff touch *only* that, or did a stray `site/` file, a `.DS_Store`, or an unrelated edit sneak in? (If `site/` shows up, your `.gitignore` from W9D1 isn't doing its job — fix that, it's exactly the kind of thing review catches.)
- **Do the `+` and `-` lines say what you expect?** Green `+` lines are additions, red `-` are removals. You should see your footer lines added and the old single-line footer removed — nothing else.
- **Would this make sense to someone else?** Read it as if you didn't write it. Is the change obvious from the diff?

Reviewing your *own* diff feels redundant the first time. It isn't: the fresh-eyes pass on a contained change is precisely where typos, accidental files, and "wait, why did I touch that" get caught — before `main`, not after.

## 5. Merge it (~10 min)

Satisfied with the diff, click **Merge pull request** → **Confirm merge**. GitHub merges `add-site-footer` into `main`. Then it offers a **Delete branch** button — click it. The branch did its one job (carry this change); now it's clutter. *One branch per change, deleted when merged* (W10D1 §3).

The merge happened *on GitHub*, so your laptop's `main` doesn't know about it yet. Bring it down:

```
git switch main
git pull
```

- **`git pull`** fetches the new state of `main` from GitHub (including your just-merged footer) and updates your local `main`. Now your laptop matches GitHub.

Clean up the local branch too — it's already deleted on GitHub:

```
git branch -d add-site-footer
```

Confirm the loop closed:

```
git log --oneline -3        # the merged footer is now on main
git branch                  # add-site-footer is gone; only main remains
open site/index.html        # rebuild if needed; footer is here on main now
```

## 6. The whole loop, named (~5 min)

You just did the complete cycle. Memorize its shape — you'll repeat it for every change from now on:

```
git switch -c <branch>      # 1. branch off main
# ...edit, build, check...
git add ...; git commit     # 2. commit on the branch
git push -u origin <branch> # 3. push the branch
# open PR on GitHub          # 4. propose
# review the diff            # 5. review
# merge + delete branch      # 6. merge
git switch main; git pull   # 7. sync local main
```

Seven steps, one change. It feels heavy for a footer. It stops feeling heavy the moment a change is risky — and tomorrow's foundations day plus Thursday's CI robot make every step earn its place.
