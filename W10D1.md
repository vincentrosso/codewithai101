## About this week — how professionals change code

For nine weeks you've made changes one way: edit a file, `git add`, `git commit`, `git push`. Straight to `main`, the single line of history, every time. It worked. But look at your own commit log this course — duplicate commits, a "day 33" pushed four times, day numbers out of order, half-finished states landing on `main` because `main` is the *only* place to land. That mess isn't carelessness; it's what *always* happens when there's nowhere to work except the live branch.

This week you learn the workflow every software team uses to avoid exactly that:

- **Today (D1):** branches — work somewhere that *isn't* `main`.
- **D2:** pull requests — propose a branch's changes, review them, merge.
- **D3:** foundations — no code, no AI — what a branch and a merge actually *are*.
- **D4:** continuous integration — a robot runs your tests on every push.
- **D5:** branch protection — make a green test suite *required* before anything merges.

The payoff lands Friday: your pytest suite stops being something you remember to run and becomes a **gate** nothing broken can pass. That's the reason Weeks 5–9 made you write tests.

Rule for the week: AI allowed, Week 4 rules. Foundations quiz D3: no AI.

## 1. Warm-up (~5 min)

```
source .venv/bin/activate
pytest -v
git status
```

Green tests, clean status (nothing uncommitted). Start from a settled `main`.

## 2. The problem with `main`, named (~10 min)

`main` is the **branch everyone trusts** — the version you'd deploy, the version that's supposed to work. When you commit directly to it, two bad things are possible:

- **You commit something half-done or broken**, and now the trusted version is broken. There's no "I'm still working on it" state — the moment you commit, it's on `main`.
- **There's no checkpoint to review.** The change *is* the history the instant you push. Nobody (including future-you) gets a chance to look at it as a proposal before it's permanent.

The fix is to do your work *somewhere else* and only bring it to `main` when it's ready and reviewed. That somewhere else is a **branch**.

## 3. What a branch is (the short version) (~10 min)

A **branch** is an independent line of work — a separate track of commits that splits off from `main` and can later be merged back. While you work on a branch, `main` sits untouched. You can make a mess, fix it, change your mind, and `main` never sees any of it until you decide to merge.

The mental picture (D3 makes it precise): `main` is a path of commits. A branch is a *fork* in that path — you step off, make a few commits on your own little side path, and later rejoin. Until you rejoin, the two paths are independent.

The everyday rule that follows: **one branch per change.** Want to add a footer to the site? Make a `add-footer` branch, do it there, merge it, delete the branch. Next change, new branch. Each branch is one self-contained unit of work — which is exactly what was missing from your straight-to-`main` history.

## 4. Make a branch and switch to it (~15 min)

Let's give this week a small, real change to carry: add a footer to the site's index page. First, create a branch and move onto it:

```
git switch -c add-site-footer
```

- **`git switch -c <name>`** — `-c` means *create*. This makes a new branch called `add-site-footer` starting from wherever `main` is now, and moves you onto it. (You may have seen `git checkout -b` do the same thing in older guides; `git switch` is the newer, clearer command for changing branches.)

Confirm where you are:

```
git branch
```

You'll see a list with a `*` next to `add-site-footer` — the `*` marks the branch you're *on*. `main` is in the list too, unchanged, waiting.

## 5. Make the change — on the branch (~15 min)

Edit `templates/index.html`. Find the footer line near the bottom and expand it into a real footer:

```html
  <p class="footer">
    <em>Generated {{ generated_at }}</em> ·
    Brand Lens evaluates how Korean brands present themselves to the US market.
  </p>
```

Rebuild and look, to be sure it's what you want:

```
python3 build_site.py
open site/index.html
```

Now commit it — exactly the commands you know, but notice *where* it lands:

```
git add templates/index.html
git commit -m "Add a descriptive footer to the site index"
```

That commit went onto `add-site-footer`, **not** `main`. Prove it to yourself:

```
git log --oneline -1            # your new commit, on this branch
git switch main
git log --oneline -1            # main does NOT have it
open site/index.html            # rebuild not needed — but main's template has no footer change
```

`main` is untouched. Your change exists only on the branch. That separation — that you *can* commit without affecting the trusted version — is the entire point. Switch back to keep working:

```
git switch add-site-footer
```

## 6. Push the branch (~10 min)

Push the branch up to GitHub. The first push of a new branch needs to tell git where it goes:

```
git push -u origin add-site-footer
```

- **`-u origin add-site-footer`** — "push to `origin` (GitHub), into a branch of the same name, and remember that pairing." After this first time, plain `git push` on this branch knows where to go.

Open your repo on GitHub in a browser. You'll see a banner: *"add-site-footer had recent pushes — Compare & pull request."* **Don't click it yet** — that's tomorrow. Today's win is that the branch is up there, visible, as a *proposal* separate from `main`.

## 7. Commit checkpoint (~5 min)

Nothing more to commit — your work is already committed on the branch and pushed. Confirm the state:

```
git status                 # clean, on add-site-footer
git branch                 # main and add-site-footer both exist
```

The "commit checkpoint" today is really a *branch* checkpoint: a clean branch pushed to GitHub, `main` untouched.
