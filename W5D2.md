## 2. What `parse_page` actually has to handle (~5 min)

Re-read `parse.py`. Notice the shape: it always returns a dict with six keys (`url`, `title`, `description`, `og_title`, `h1`, `canonical`). Any field can be `None` if the corresponding tag is missing.

The cases the function quietly handles:

1. A page with no `<title>` at all.
2. A page with `<title></title>` (empty title).
3. A page with no `<meta name="description">`.
4. A page with multiple `<h1>` tags (it picks the first).
5. A page with no `<link rel="canonical">`.
6. A page whose HTML is malformed (BeautifulSoup is lenient — most "malformed" HTML still parses).

Each case is a sentence that `parse_page` makes a promise about. Each sentence is a test waiting to be written.

def test_parse_page_returns_none_when_title_missing():

## 4. The DRY problem (~5 min)

Look at your test file. Three of the four tests build an HTML string in the Arrange block. The HTML is short here, but it's already getting repetitive: each test repeats `<html><head>...</head><body></body></html>` scaffolding.

When the same setup shows up in multiple tests, two things go wrong:

- **Noise.** You skim the test file looking for the assertion and your eyes have to wade through identical setup each time.
- **Drift.** When you eventually want to change the scaffolding (say, add a `<meta charset="utf-8">`), you have to remember to change it in every test.

The fix is a **fixture** — a chunk of setup pytest builds for you, named, that you ask for by adding it as a function parameter.

## 5. conftest.py and @pytest.fixture (~15 min)

Make a new file in `tests/`:

```
touch tests/conftest.py
subl tests/conftest.py
```

```python
import pytest


@pytest.fixture
def full_page_html():
    return """
    <html>
      <head>
        <title>Brand Lens</title>
        <meta name="description" content="A tool for evaluating brand presence">
        <meta property="og:title" content="Brand Lens — OG">
        <link rel="canonical" href="https://example.com/canonical">
      </head>
      <body>
        <h1>Welcome to Brand Lens</h1>
        <h1>Second h1, should be ignored</h1>
      </body>
    </html>
    """
```

Two new ideas:

- **`@pytest.fixture`** is a decorator that says "this function produces a value that tests can ask for." The function name (`full_page_html`) is the fixture's name.
- **`conftest.py`** is a special filename pytest looks for in the `tests/` directory. Any fixture defined here is available to every test file in that directory automatically — no import needed.

Now use it. Open `tests/test_parse.py` and add a new test below the others:

```python
def test_parse_page_extracts_og_title(full_page_html):
    record = parse_page(full_page_html, "https://example.com/")
    assert record["og_title"] == "Brand Lens — OG"
```

Read the function signature. The parameter `full_page_html` matches the fixture name in `conftest.py`. When pytest runs this test, it sees the parameter, looks up the fixture by name, calls the fixture function, and passes the returned value as the argument. You never call the fixture yourself — pytest wires it up.

Run:

```
pytest -v
```

Five tests passing. The new test's Arrange block has shrunk to zero lines.

## 6. Two more tests using the fixture (~10 min)

Add these to `tests/test_parse.py`:

```python
def test_parse_page_picks_first_h1(full_page_html):
    record = parse_page(full_page_html, "https://example.com/")
    assert record["h1"] == "Welcome to Brand Lens"


def test_parse_page_extracts_canonical(full_page_html):
    record = parse_page(full_page_html, "https://example.com/")
    assert record["canonical"] == "https://example.com/canonical"
```

Both ask for the same fixture. pytest builds the HTML once per test (technically once per call — fixtures default to function scope). No setup duplication, no drift risk.

```
pytest -v
```

Seven tests, all green.

## 7. A test is a tiny spec (~5 min)

Look at your test file from top to bottom. Read just the function names:

- `test_parse_page_extracts_title`
- `test_parse_page_returns_none_when_title_missing`
- `test_parse_page_extracts_meta_description`
- `test_parse_page_returns_none_when_description_missing`
- `test_parse_page_extracts_og_title`
- `test_parse_page_picks_first_h1`
- `test_parse_page_extracts_canonical`

Without opening `parse.py`, that list tells you what the function does and how it handles missing data. That's not an accident — that's tests doing double duty as **documentation that can never go out of date**. If a behavior listed there changes, the test fails and you find out immediately. Plain prose documentation doesn't have that property.

This is the part of testing that's hardest to feel until you've lived with a test suite for a while: the suite becomes your authoritative description of what the code does. Code without tests is a rumor.
