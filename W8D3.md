## 2. The one idea: separation of concerns (~15 min)

A **concern** is one job, one reason a piece of code exists. **Separation of concerns** is the principle that each piece should have *one* concern, and different concerns should live in different pieces. You've applied it repeatedly:

- **W3D2:** you split one script into `fetch.py` (get the bytes off the network), `parse.py` (turn bytes into fields), and `runner.py` (orchestrate). Three concerns, three files.
- **W5D3:** you learned that pure functions (decide, transform) are easy to test and side-effectful ones (fetch, write, call API) are hard — and that a good codebase pushes side effects to the *edges* and keeps the interior pure. That's separation of concerns: "doing I/O" is a different concern from "computing the answer."
- **W6:** `db.py` holds *all* the database concern. Nothing else in the project writes SQL. If you switch to a different database later, one file changes.
- **W8D1:** the *template* holds how the brief looks; the *code* holds the data. Presentation is a different concern from logic. And `render_brief` (pure, builds the string) is split from `write_brief` (side effect, touches disk).

Why it pays off, concretely:

- **You can find things.** A bug in how a page is parsed is in `parse.py`. A bug in how the brief looks is in the template. You don't hunt through a 500-line file.
- **You can change one thing without breaking others.** Reword the report → edit the template, and `db.py` can't possibly break, because it never knew about the report.
- **You can test the pieces.** Pure pieces with one job are testable in isolation (the entire reason Week 5 worked). A function that fetches *and* parses *and* writes is testable only as a whole.
- **You can replace a piece.** Swap `requests` for another HTTP library? Only `fetch.py` cares.

The smell of a *violated* concern: a function whose name needs the word "and" — `fetch_and_parse_and_save`. That's three concerns wearing one coat.

## 3. The anatomy of brand-lens right now (~15 min)

Look at your project as a whole. It's roughly:

```
brand-lens/
├── runner.py          # orchestrates: config → fetch → parse → store
├── fetch.py           # concern: get a page off the network
├── parse.py           # concern: HTML → fields
├── db.py              # concern: everything SQLite
├── brief.py           # concern: ask the LLM, get a structured brief
├── render.py          # concern: context → markdown string, and write it
├── generate_briefs.py # orchestrates: db → context → render → briefs/
├── brands.yaml        # config: which brands, which pages
├── templates/
│   └── brief.md       # concern: how a brief looks
├── tests/             # concern: verify the above
│   ├── conftest.py
│   ├── test_parse.py
│   ├── test_fetch.py
│   ├── test_db.py
│   └── test_brief.py
├── .env               # secrets (gitignored)
├── .gitignore
├── brand_lens.db      # data (gitignored)
└── briefs/            # output (gitignored)
```

Read it top to bottom. Every file has a one-sentence job — that's not luck, it's the result of seven weeks of splitting things when they got too big. Notice the three *categories*:

- **Code** (`*.py`, `templates/`) — committed. This *is* the project.
- **Config** (`brands.yaml`) — committed. Inputs you control.
- **Secrets, data, and output** (`.env`, `brand_lens.db`, `briefs/`) — gitignored. Generated or sensitive; not the project, just what the project produces or needs.

That last split is a concern too: *the code is permanent and shared; the data is local and disposable.* Mixing them — committing the database, say — is a category error that bloats the repo and leaks data.

## 4. Dependencies and reproducibility (~15 min)

Your code imports `requests`, `beautifulsoup4`, `pyyaml`, `pytest`, `anthropic`, `python-dotenv`, `jinja2`. None of those are *your* code — you `pip install`ed them into your venv. They're **dependencies**: code your project needs but doesn't contain.

Here's the problem they create. If Vincent clones your repo onto his laptop, he gets your `.py` files — but *not* the libraries, because those live in your venv (which isn't committed) on Anthropic's servers, not in git. He runs `python3 runner.py` and gets `ModuleNotFoundError: No module named 'requests'`. Your project doesn't run on his machine. "Works on my machine" is the oldest problem in software.

The fix is a **`requirements.txt`** — a plain list of every dependency, committed to the repo. Anyone who clones it runs `pip install -r requirements.txt` and gets the exact same libraries. The file is the project's recipe.

Two refinements you'll apply tomorrow:

- **Pinning.** `requests` alone means "any version." `requests==2.32.3` means "this exact version." Pinning matters because libraries change: a version that works today might behave differently in a year. Pinning makes the project **reproducible** — same versions, same behavior, regardless of when or where it's installed.
- **Only what you import.** `requirements.txt` should list what the project actually uses, not every package that happens to be in your venv. A bloated list installs things you don't need and hides what you do.

This is the same instinct as the `.env.example` from W7: give the next person a recipe for the environment, without making them reverse-engineer it.

## 5. The README — the front door (~10 min)

Code tells *how*. A **README** tells a human *what* and *why* and *how do I start*. It's the first file anyone opens — including you in six months, having forgotten everything. A good project README answers, in order:

1. **What is this?** One or two sentences. ("Brand Lens evaluates a Korean brand's US web presence and generates a per-brand brief.")
2. **How do I set it up?** Clone, make a venv, `pip install -r requirements.txt`, copy `.env.example` to `.env` and add a key.
3. **How do I run it?** The actual commands: `python3 runner.py`, `python3 generate_briefs.py`.
4. **How is it laid out?** A short version of section 3 — where things live.

That's it. Not a novel. The test of a README: could someone who's never seen the project get it running *without asking you*? If yes, it's doing its job. (You've been reading exactly this kind of README all course — the lessons' "what done looks like" sections are a cousin of it.)
