*v1.0.0*

**Goal for today:** put the site on the web. By end of day you paste a link into a browser — on your phone, even — and Brand Lens loads from the internet.


You should see your IAM user's identity and a freshly built `site/`. Today that folder goes live.

## 2. What a bucket is (~5 min)

An **S3 bucket** is a named container for files (S3 calls them *objects*) living on AWS's storage. Think of it as a folder in the cloud with a globally-unique name and a web address. Two things make it perfect for your static site:

- It just *holds files* and hands them out — exactly what a static site needs and nothing more (W9D3). No server, no backend.
- With one setting flipped, a bucket can act as a **website**: point a browser at it and it serves `index.html` by default — which is why you named your front door `index.html` back in W9D2.

Bucket names are **globally unique across all of AWS** — like a username nobody else on Earth can already have. So you'll want something specific.

## 3. Create the bucket (~15 min)

Pick a unique name. A good pattern is your project plus something distinctive (initials, a number). Throughout below, replace `brand-lens-tyler-2026` with *your* chosen name:

```
aws s3 mb s3://brand-lens-tyler-2026
```

- **`aws s3 mb`** — "make bucket" (`mb`, like `mkdir`). `s3://name` is how you address a bucket.

If it complains the name is taken, pick another — globally unique means *globally*. On success it prints `make_bucket: brand-lens-tyler-2026`. Confirm:

```
aws s3 ls
```

Your bucket in the list. It exists, it's empty, and right now it's completely private — nobody but you can see anything in it. The next steps put files in and open it up *deliberately*.

## 4. Copy the site up (~15 min)

`aws s3 sync` copies a local folder into a bucket, uploading what's new or changed — the cloud cousin of `git push`, but for files:

```
aws s3 sync site/ s3://brand-lens-tyler-2026
```

It prints an `upload:` line per file — your brand pages, `index.html`, `style.css`. Confirm they landed:

```
aws s3 ls s3://brand-lens-tyler-2026
```

The files are in the cloud. But if you tried to view them right now, you'd be denied — a fresh bucket is private by default (a *good* default; most buckets shouldn't be public). A public website is a choice you make explicitly, next.

## 5. Turn on website hosting (~10 min)

Tell the bucket to behave like a website — serve `index.html` for the root, and use it for missing pages too:

```
aws s3 website s3://brand-lens-tyler-2026 --index-document index.html --error-document index.html
```

- **`--index-document index.html`** — when someone visits the bucket's root, serve `index.html` (your front door — this is why the filename mattered).
- **`--error-document index.html`** — for a not-found path, fall back to the index rather than an ugly error.

This flips on the website *behavior*, but the files are still private. One more deliberate step opens them.

## 6. Open public read access (~20 min)

This is the step to do slowly and understand, because you're intentionally making files public — the one moment all week where "locked down" gives way to "the whole world can read this." Two parts.

**First, allow the bucket to be public at all.** New buckets ship with a master "Block Public Access" switch on, as a safety net. For a public website you turn it off *for this bucket*:

```
aws s3api put-public-access-block \
  --bucket brand-lens-tyler-2026 \
  --public-access-block-configuration "BlockPublicAcls=false,IgnorePublicAcls=false,BlockPublicPolicy=false,RestrictPublicBuckets=false"
```

**Second, grant the public *read* — and only read.** A **bucket policy** is a rule saying who may do what. This one says "anyone may read (`GetObject`) the files, and nothing else." Write it to a file so it's reviewable (and reusable tomorrow):

```
subl bucket-policy.json
```

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadForWebsite",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::brand-lens-tyler-2026/*"
    }
  ]
}
```

Read it — you can, it's just labels:

- **`"Effect": "Allow"`** — this rule grants (vs `Deny`).
- **`"Principal": "*"`** — *who*: `*` means everyone, the public. (This is the line that makes it a public website. It belongs *only* on a bucket you mean to publish.)
- **`"Action": "s3:GetObject"`** — *what*: read an object. Read only — not write, not delete, not list. The public can *view* your pages, not change them. Least privilege again, now pointed outward.
- **`"Resource": "...:::brand-lens-tyler-2026/*"`** — *which* files: everything in this one bucket (`/*`). Use *your* bucket name here.

Apply it (use your real bucket name in both the policy and the command):

```
aws s3api put-bucket-policy --bucket brand-lens-tyler-2026 --policy file://bucket-policy.json
```

## 7. Visit it — the payoff (~5 min)

Your site's web address (the **website endpoint**) has this shape — fill in your bucket name and the region you configured on D1:

```
http://brand-lens-tyler-2026.s3-website-us-east-1.amazonaws.com
```

(If you chose a region other than `us-east-1`, the host changes to match it — e.g. `s3-website-us-west-2`.) Open it in a browser. **Brand Lens loads from the internet.** Click the index into a brand and back — the relative links (W9D3) work unchanged, because the pages are siblings in the bucket just as they were siblings on disk. Open it on your phone. Send the link to someone. It's real, and it's public.

Note the `http://`, not `https://` — an S3 website endpoint is plain HTTP. That's fine for a public brief with no logins or secrets; the lock-icon `https://` version needs an extra AWS service (CloudFront) that's a Course-2 topic. The foundations day names why.

## 8. Commit (~5 min)

You created one file worth keeping — the bucket policy (a reusable, reviewable artifact). Ship it the proper way, through the workflow you built last week:

```
git switch -c add-bucket-policy
git add bucket-policy.json
git commit -m "Add S3 public-read bucket policy for the static site"
git push -u origin add-bucket-policy
```

Open the PR, watch CI go green, merge, delete the branch, `git pull`. (Yes — even a one-file change goes through a branch and the gate now. That's the Week 10 habit, and it's not optional anymore: branch protection won't let you push to `main` directly.)
