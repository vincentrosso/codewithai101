## 2. Demo — walk Vincent through the database (~10 min)

When Vincent joins:

- Show the schema (`.schema` in the `sqlite3` tool). Explain the two tables and *why* they're two: the `brands`/`pages` split and the foreign key. This is the D3 normalization idea, in your own words.
- Run your three prepared queries live. Read each result.
- Run the pipeline and show a row landing in the database. Then run it *again* and show the row count didn't grow — demonstrate the upsert.

You're aiming to explain *why* the schema is shaped the way it is and *why* re-runs don't duplicate — not to recite SQL keywords.

## 3. The break-and-catch exercise (~15 min)

The meat of the review. Vincent introduces a small, plausible change to `db.py` while you look away — the kind Claude Code might propose in a future diff:

- Change `upsert_page`'s `INSERT OR REPLACE` back to a plain `INSERT` (re-runs would crash or duplicate).
- Swap two columns in the `upsert_page` value tuple (e.g. `title` and `description`) so data lands in the wrong column.
- Change `pages_for_brand`'s `WHERE brand_slug = ?` to `WHERE brand_slug = brand_slug` (matches every row — a real bug that looks almost right).
- Drop the `UNIQUE` from the `url` column in `SCHEMA`.

You then run:

```
pytest -v
```

**Outcome A — a test caught it.** Read the failure aloud. Which test fired, and why? Revert, re-run, confirm green.

**Outcome B — no test caught it.** The more useful case: the suite has a gap. Together:

1. Name the gap — which behavior wasn't being verified?
2. Write the test that would have caught it, using the `db_conn` fixture.
3. Have Vincent re-introduce the bug; re-run; the new test should fail (proving it checks what you think).
4. Revert; re-run; the new test passes.

Note which breaks your suite caught and which it missed. The swapped-column break (`title`/`description`) is a good one to reflect on: does any test actually assert that the title comes back in the *title* column, or only that *a* row comes back? That gap is easy to have.

## 4. Claude judgment — let Claude Code write a query (~15 min)

Open Claude Code:

```
claude
```

Ask:

> Add a function to db.py called `count_pages_with_description(conn)` that returns the number of pages whose description is not null. Show me the code as a diff but don't apply it yet.

Read the diff carefully (W4D3 rules). Then judge it against everything this week taught:

1. **Is the SQL correct?** Does it use `IS NOT NULL` — or did it write `!= NULL` or `= NULL`, which (D2) returns the wrong answer? This is the single most common SQL mistake and a great test of whether Claude got it right *here, today*.

2. **Does it use a `?` parameter for any value, or does it build SQL with an f-string?** This function takes no user value, so it may not need one — but ask Claude a follow-up: *"add a version that counts pages for a given brand_slug passed as an argument."* Now there's a value. Does Claude use `?`, or does it f-string the `brand_slug` into the query string? If it f-strings it, that's the SQL-injection footgun from D2 — reject or fix it, and say why in your journal.

3. **Does it return what you'd expect?** `COUNT(*)` via `fetchone()` comes back as a one-element tuple like `(7,)`, not the bare number `7`. Does Claude's code unpack it (`cursor.fetchone()[0]`), or would the caller get a tuple by surprise?

4. **Is there a test?** If Claude offered a test, does it use the `db_conn` in-memory fixture, or does it write to the real `brand_lens.db`? A db test that touches the real file is a flaky test waiting to happen (W5D3 §7).

Decide: accept, reject, or modify — and write the reason in your journal. The point isn't to merge the function. It's to judge Claude's SQL against the standards you built this week, especially the `IS NOT NULL` and `?`-parameter ones, which are exactly the things a confident-sounding wrong answer gets wrong.
