## About this week — why a database, why now

Through Week 3 your pipeline wrote `results.json` and a flat CSV. That was the right call then — a file is the simplest thing that works, and you could open it and read it. But you've outgrown it, and here's the specific way:

Ask your current setup a question. *"How many pages across all brands have a meta description?"* With JSON, you'd write a script: open the file, parse it, loop over brands, loop over pages, count the ones where `description` isn't `None`. Ten lines. Now ask *"which brands have a page whose title is empty?"* — another script. *"What's the average number of pages per brand?"* — another. Every question is a new little program.

A database flips that. You define the shape of your data **once** (the schema), and then any question is a **query** — usually one line. "How many pages have a description" becomes:

```sql
SELECT COUNT(*) FROM pages WHERE description IS NOT NULL;
```

That's the trade this week is about: a bit of structure up front buys you a query language that answers questions you haven't thought of yet.

## 2. What SQLite is (~5 min)

**SQLite** is a database that lives in a single file. No server to start, no install, no account. It's built into Python's standard library as the `sqlite3` module — the same way `json` and `csv` are. The entire database — every table, every row — is one file on your disk, here `brand_lens.db`.

That makes it perfect for this project: you get a real database with real SQL, but it's as portable as a file. You can copy it, commit it (we won't — data files don't belong in git), delete it and start over. Bigger systems use Postgres or MySQL, which *do* run as servers; the SQL you learn here transfers almost unchanged.

You'll talk to SQLite two ways this week:

- From **Python**, with the `sqlite3` module — that's how the pipeline will write to it.
- From the **command line**, with the `sqlite3` tool — that's how *you* will poke at it to check your work.

## 3. The schema (~15 min)

A **schema** is the shape of your data: what tables exist, and what columns each table has. You decide it up front. Today's schema is two tables.

Create the file:

```
touch db.py
subl db.py
```

```python
import sqlite3

DB_PATH = "brand_lens.db"

SCHEMA = """
CREATE TABLE IF NOT EXISTS brands (
    slug TEXT PRIMARY KEY,
    name TEXT
);

CREATE TABLE IF NOT EXISTS pages (
    id          INTEGER PRIMARY KEY,
    brand_slug  TEXT NOT NULL REFERENCES brands(slug),
    url         TEXT NOT NULL UNIQUE,
    status      INTEGER,
    title       TEXT,
    description TEXT,
    og_title    TEXT,
    h1          TEXT,
    canonical   TEXT,
    error       TEXT,
    fetched_at  TEXT NOT NULL
);
"""


def get_connection(db_path=DB_PATH):
    return sqlite3.connect(db_path)


def init_db(db_path=DB_PATH):
    conn = get_connection(db_path)
    conn.executescript(SCHEMA)
    conn.commit()
    conn.close()
```

Read it carefully — there's a lot of new vocabulary in a small space:

- **Two tables.** `brands` is tiny: one row per brand, identified by its `slug` (the short name like `innisfree` you already use as the key in `brands.yaml`). `pages` is the workhorse: one row per fetched page, with a column for each field `parse_page` produces, plus `status` and `error` from `fetch_one`.
- **`PRIMARY KEY`** — the column that uniquely identifies a row. No two brands can share a `slug`; no two pages can share an `id`. `INTEGER PRIMARY KEY` is special in SQLite: it auto-assigns a number to each new row, so you never set `id` yourself.
- **`REFERENCES brands(slug)`** — this is a **foreign key**: it says "the `brand_slug` in a `pages` row points at a real `slug` in the `brands` table." It's how the two tables are wired together. (We'll lean on this on D2's `JOIN` and explain it properly on D3.)
- **`NOT NULL`** — this column is never allowed to be empty. A page must have a `brand_slug`, a `url`, and a `fetched_at`. But `title` can be `NULL` (some pages have no `<title>` — you wrote the test for that in Week 5).
- **`UNIQUE`** on `url` — no two rows can have the same URL. This matters on D4: it's what lets a re-run *update* a page instead of inserting a duplicate.
- **`CREATE TABLE IF NOT EXISTS`** — run `init_db()` ten times and it won't error on runs 2–10. The table's already there.
- **`executescript`** — runs a string containing multiple SQL statements (we have two `CREATE TABLE`s). For single statements you'll use `execute`, coming up next.

## 5. Insert a few rows by hand (~15 min)

Add a function to `db.py`:

```python
def insert_brand(conn, slug, name):
    conn.execute(
        "INSERT OR IGNORE INTO brands (slug, name) VALUES (?, ?)",
        (slug, name),
    )


def insert_page(conn, brand_slug, record):
    conn.execute(
        """
        INSERT INTO pages
            (brand_slug, url, status, title, description,
             og_title, h1, canonical, error, fetched_at)
        VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
        """,
        (
            brand_slug,
            record["url"],
            record.get("status"),
            record.get("title"),
            record.get("description"),
            record.get("og_title"),
            record.get("h1"),
            record.get("canonical"),
            record.get("error"),
            record.get("fetched_at"),
        ),
    )
```

The single most important thing on this page: **those `?` marks.**

```python
conn.execute("INSERT ... VALUES (?, ?)", (slug, name))
```

You do **not** build SQL by gluing strings together. You write the SQL with `?` placeholders, and pass the actual values as a separate tuple. SQLite fills them in safely. The wrong way — `f"INSERT ... VALUES ('{slug}', '{name}')"` — looks like it would work, and it does until a value contains a quote or something hostile, at which point your query breaks or becomes a security hole (it has a famous name: **SQL injection**). The `?` placeholder is not a style preference. It is the rule. We'll come back to it on D2.

Now insert some real data. Make a throwaway script:

```
touch seed_demo.py
subl seed_demo.py
```

```python
import db

conn = db.get_connection()

db.insert_brand(conn, "innisfree", "Innisfree")
db.insert_brand(conn, "laneige", "Laneige")

db.insert_page(conn, "innisfree", {
    "url": "https://www.innisfree.com/",
    "status": 200,
    "title": "Innisfree — Natural Korean Skincare",
    "description": "Clean beauty from Jeju Island",
    "og_title": "Innisfree",
    "h1": "Welcome",
    "canonical": "https://www.innisfree.com/",
    "error": None,
    "fetched_at": "2026-05-27T10:00:00",
})

db.insert_page(conn, "laneige", {
    "url": "https://www.laneige.com/",
    "status": 200,
    "title": "LANEIGE",
    "description": None,
    "og_title": "LANEIGE US",
    "h1": "Water Sleeping Mask",
    "canonical": "https://us.laneige.com/",
    "error": None,
    "fetched_at": "2026-05-27T10:00:01",
})

conn.commit()
conn.close()
print("seeded")
```

Run it:

```
python3 seed_demo.py
```

You should see `seeded`. One brand row each, one page row each — written to disk.

**`conn.commit()` is load-bearing.** SQLite holds your inserts in a pending transaction until you commit. Forget the `commit()` and the rows vanish when the connection closes. (This is the same "did it actually save?" worry as forgetting to close a file in Week 3 — same lesson, new tool.)

## 6. Read it back from the command line (~10 min)

Now look at the data with the `sqlite3` command-line tool. This is separate from Python — it's a program for poking at the database directly.

```
sqlite3 brand_lens.db
```

You're now at a `sqlite>` prompt. Try:

```sql
.tables
```

Lists your tables: `brands  pages`. The commands starting with a dot (`.tables`, `.schema`, `.quit`) are tool commands, not SQL.

```sql
.schema pages
```

Prints the `CREATE TABLE` for `pages` — the schema you wrote, read back from the file.

Now real SQL. **Predict the output before you hit Enter** — write your guess in your journal:

```sql
SELECT * FROM brands;
SELECT title, description FROM pages;
```

Did the Laneige `description` come back empty? It should — you inserted `None`, which SQLite stores as `NULL` and the tool shows as blank. (If your prediction was wrong, edit your journal and say what surprised you — that habit is the point, not a slip.)

Leave the tool:

```sql
.quit
```

## 7. The mental shift (~5 min)

Look at what just happened. You defined the data's shape once in `db.py`. Then from a completely different program — the `sqlite3` CLI — you asked it questions. The data outlived the script that wrote it, and it's queryable by anything that can open the file.

That's the difference from `results.json`. The JSON was inert: a script wrote it, and only a script that knows its exact shape can read it. The database is *active*: it knows its own shape (the schema), enforces rules (`NOT NULL`, `UNIQUE`), and answers questions in a language built for asking them. Tomorrow you'll learn that language properly.
