## 2. What an index page is (~5 min)

Yesterday's brand page has a dead link at the top: `<a href="index.html">← All brands</a>`. Today you build what it points to. An **index page** (`index.html`) is a site's front door — the page you land on first, whose job is to send you to the others. Yours will be a list of brands, each a link to that brand's brief page.

There's a convention hiding here that pays off in Week 11: a web server, handed a bare directory, looks for a file literally named `index.html` and serves *that* by default. It's why `example.com/` shows you a page without naming a file. Name your front door `index.html` and the web does the right thing for free.

## 3. A query that lists the brands (~15 min)

The index needs every brand's name and slug. You have a `db.py` that owns all SQL (W6) — add the helper there, not inline. Open `db.py` and add:

```python
def all_brands(conn):
    """Return [{'slug': ..., 'name': ...}, ...] for every brand, name-sorted."""
    cur = conn.execute(
        "SELECT slug, name FROM brands ORDER BY name"
    )
    return [dict(row) for row in cur.fetchall()]
```

(If your `brands` table calls the display column something other than `name`, match it. The `dict(row)` step is the same one `page_fields_for_brand` uses — turning each row into a plain dict so the template can say `brand.name`.)

This is separation of concerns again (W8D3): the index template will ask "what brands are there?" and `db.py` answers. The template never sees SQL.

## 4. The index template (~20 min)

```
subl templates/index.html
```

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>Brand Lens — US Presence Briefs</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>
  <h1>Brand Lens</h1>
  <p>US market-presence briefs, one per brand.</p>

  <ul class="brand-list">
    {% for brand in brands %}
    <li><a href="{{ brand.slug }}.html">{{ brand.name }}</a></li>
    {% else %}
    <li>No brands yet.</li>
    {% endfor %}
  </ul>

  <p class="footer"><em>Generated {{ generated_at }}</em></p>
</body>
</html>
```

Two things worth naming:

- **`<a href="{{ brand.slug }}.html">`** builds each link by gluing the slug to `.html` — so brand `innisfree` links to `innisfree.html`, the exact file you wrote yesterday. A *relative* link again: "the file next to me named `innisfree.html`." This is why every page lives in the same `site/` folder — relative links between siblings just work.
- **`{% for ... %} ... {% else %} ... {% endfor %}`** — yes, a `for` loop can have an `else`. In `jinja2` the `{% else %}` of a *loop* runs when there's nothing to loop over. So an empty database renders "No brands yet" instead of a blank list. (That's a different `else` from the `{% if %}/{% else %}` you used yesterday — same word, different host. Note it; D3's cousin foundations days live on exactly these small ambiguities.)

## 5. A stylesheet (~20 min)

Right now the pages are black Times-New-Roman on white — technically the web, spiritually a ransom note. A **stylesheet** (CSS — Cascading Style Sheets) is a separate file that says how the HTML should *look*: fonts, colors, spacing. It's the same separation-of-concerns move as templates: HTML is the content and structure, CSS is the appearance. Keep them apart and you can restyle the whole site without touching a single template.

Put it in the templates' neighborhood so the build can copy it:

```
mkdir -p static
subl static/style.css
```

```css
body {
  font-family: -apple-system, system-ui, sans-serif;
  max-width: 720px;
  margin: 2rem auto;
  padding: 0 1rem;
  color: #222;
  line-height: 1.5;
}

h1 { font-size: 1.8rem; }
h2 { font-size: 1.2rem; margin-top: 1.5rem; color: #444; }

a { color: #1a5fb4; }

.brand-list { list-style: none; padding: 0; }
.brand-list li { padding: 0.4rem 0; border-bottom: 1px solid #eee; }

.footer { color: #888; font-size: 0.85rem; margin-top: 2rem; }
```

You don't need to understand every line of CSS to use it — read it as "selector, then a block of properties." `body { ... }` styles the whole page; `h2 { ... }` styles every `<h2>`; `.brand-list { ... }` styles anything with `class="brand-list"` (that's what the `class=` attributes in the templates are for — handles CSS can grab onto). Tweak a number, rebuild, look. This is the one place in the course where eyeballing-and-adjusting beats understanding-first; treat it as a knob, not a theorem.

Now link the stylesheet from **both** templates' `<head>` so every page uses it. The index already has it (step 4). Add the same line to `templates/brief.html`, just under its `<title>`:

```html
  <link rel="stylesheet" href="style.css">
```

`<link rel="stylesheet" href="style.css">` tells the browser: "while drawing this page, also fetch `style.css` from next to me and apply it." Another relative reference — which means `style.css` has to end up *in* `site/` alongside the pages. You'll make the build copy it there tomorrow; for now, copy it by hand so you can see the effect:

```
cp static/style.css site/style.css
open site/innisfree.html
```

The same page, now with margins, a real font, and breathing room. Small change, big difference in whether it looks like *a thing*.

## 6. Render the index by hand (~10 min)

Tomorrow's `build_site.py` will do this properly. For now, prove the index renders. Add a renderer to `render.py`:

```python
def render_index(brands, generated_at):
    """Render the index page listing all brands. Pure — no I/O."""
    template = env.get_template("index.html")
    return template.render(brands=brands, generated_at=generated_at)
```
