## 2. A table is a list of dicts you already know (~10 min)

You've been working with this shape since Week 2:

```python
pages = [
    {"url": "https://a.com", "title": "A", "status": 200},
    {"url": "https://b.com", "title": "B", "status": 404},
]
```

A database **table** is that exact idea, made strict and stored on disk:

- The **list** is the table (`pages`).
- Each **dict** is a **row**.
- Each **key** is a **column**.

The strictness is the difference. In your Python list, one dict could have a `title` key and the next could forget it — nothing stops you. A table's **schema** says every row has exactly these columns, with these types, and these rules (`NOT NULL`, `UNIQUE`). The schema is a *contract*: the database refuses to store a row that violates it. That refusal is a feature — it catches a whole class of bug ("oops, I forgot to set the url") at write time instead of letting it rot in your data.

So: a table is a list of dicts where the database enforces that every dict has the same shape.

## 3. Primary keys — how you name one row (~10 min)

If a table is a list of rows, how do you point at *one specific* row? Not "the third one" — rows have no reliable order. You need a column whose value is unique to each row, so naming the value names the row. That column is the **primary key**.

- In `brands`, the primary key is `slug`. `innisfree` refers to exactly one brand row, forever.
- In `pages`, the primary key is `id` — an integer SQLite assigns automatically. You didn't have a natural unique field to use (two pages could share a title), so you let the database mint an arbitrary unique number.

Two styles, both common:

- A **natural key** — a real piece of data that happens to be unique (`slug`).
- A **surrogate key** — a meaningless auto-number whose only job is to be unique (`id`).

The primary key is also a promise the database enforces: try to insert a second brand with `slug = 'innisfree'` and it errors. That's `PRIMARY KEY` doing its job — guaranteeing you can always identify a row unambiguously.

## 4. Foreign keys — how tables connect (~10 min)

Here's the question that justifies having two tables. A page belongs to a brand. How do you record that?

The bad answer: copy the brand's whole name into every page row. Then "Innisfree" is written into eight page rows. Now the company rebrands to "innisfree by Amorepacific" and you have to find and fix eight rows — and if you miss one, your data now disagrees with itself. That's **duplication**, and duplication is where data goes wrong.

The good answer: store the brand once in `brands`, and in each `pages` row store only a *reference* to it — the brand's primary key. That reference is the **foreign key**:

```
pages.brand_slug  ──points at──>  brands.slug
```

`brand_slug` in a page row isn't the brand's name; it's a pointer to the one brand row that has the name. The name lives in exactly one place. Rebrand? Change one row in `brands`, and every page that points at it now reflects the change, because they never stored a copy in the first place.

The `JOIN` you wrote on D2 is how you *follow* that pointer — "for each page, go find the brand row its `brand_slug` points at, and bring the name along." The foreign key is the connection; the join is the act of using it.

This idea — *store each fact once, point at it from everywhere else* — is the core of what people call **normalization**. You don't need the formal theory. You need the instinct: when you notice the same value copied across many rows, that value probably wants its own table.

## 5. Why not one big JSON blob? (~10 min)

You might reasonably ask: I already had `results.json`, with brands containing pages, nested neatly. Why is that worse than two tables?

It isn't worse for *storing*. It's worse for *asking*. Three specific ways:

1. **Querying.** "Pages with no description, across all brands, sorted by brand" is one line of SQL and a custom loop in JSON-land. Every new question in JSON is new code; in SQL it's a new query.
2. **Rules.** The JSON file can't stop you from writing a page with no URL, or two pages with the same URL, or a page pointing at a brand that doesn't exist. The schema enforces all three. The file enforces nothing.
3. **Updating one thing.** Change a single page's title in a 2,000-line JSON file: you load the whole file, find the spot, edit, write the whole file back. In SQL it's one `UPDATE ... WHERE id = ?`. The database changes one row; it doesn't rewrite everything.

The trade is real, not free: the schema is rigid. Adding a new field means altering the table, not just adding a key to a dict. For data you'll *query and update over and over* — which is exactly what brand-lens data is — that rigidity buys far more than it costs. For a one-off scratch file, JSON is still the right tool. Match the tool to the job.

## 6. NULL is not zero and not empty string (~10 min)

`NULL` tripped you on D2 (`WHERE description = NULL` returned nothing) and it's worth understanding, not just memorizing.

`NULL` means **"there is no value here / unknown."** It is specifically *not*:

- `0` — that's a known number.
- `""` — that's a known string that happens to be empty.
- `False` — that's a known boolean.

A page with `description IS NULL` had no `<meta name="description">` tag at all. A page with `description = ''` had the tag, but it was empty. Those are different facts, and the schema lets you tell them apart.

The reason `description = NULL` returns nothing: SQL treats `NULL` as "unknown," and *any* comparison with an unknown is itself unknown — not true. `unknown = unknown` is unknown, so the row doesn't match. To ask about nullness you need the special operators `IS NULL` / `IS NOT NULL`, which are built for exactly this. This is the same distinction as `None` vs `0` vs `""` in Python — you already met it when `parse_page` returned `None` for a missing title rather than `""`.

## 7. A note on scale and indexes (~5 min)

One thing you don't need yet but should have heard of. When a table gets large and you query `WHERE url = ?` a lot, the database can either scan every row looking for matches (slow) or keep a sorted lookup structure — an **index** — that jumps straight to the match (fast). A `PRIMARY KEY` and a `UNIQUE` column get an index automatically, which is part of why looking up a page by its URL on D4 will be fast. You won't create indexes by hand in this course; just know the word and that it's the answer to "why is this query slow on a big table."
