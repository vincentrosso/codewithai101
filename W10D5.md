# Week 10, Day 5 — Friday Review: Make the Gate Required
*v1.0.0*

**Goal for today:** close the loop on the whole workflow. You'll turn on **branch protection** so a green CI check is *required* before anything can merge to `main` — making the gate real, not optional. Then demo a PR getting blocked by a red suite, run a CI-based break-and-catch, judge a Claude-written workflow change, and write the Week 10 retrospective.

**Time budget:** ~30 min mentor call, ~45 min hands-on, ~25 min retrospective journal, ~10 min slack.

## 1. Before the call — turn on branch protection (~15 min)

Right now CI *runs*, but a red check doesn't actually *stop* a merge — you could still click "Merge" past a red ✗. Branch protection fixes that: it makes GitHub *refuse* to merge a PR whose checks haven't passed. The gate goes from advisory to enforced.

On GitHub: **Settings → Branches → Add branch ruleset** (or "Add rule" on older UIs). Then:

- **Branch name pattern:** `main`.
- Enable **Require status checks to pass before merging**, and in the search box add your check (**CI / test**).
- (Recommended) Enable **Require a pull request before merging** — this forbids pushing *directly* to `main` at all, so every change must go through a PR. That's the straight-to-`main` habit, finally locked out by the repo itself.

Save. You've just made the rule structural: `main` now *only* accepts changes that went through a PR and passed CI. The discipline no longer depends on you remembering it.

## 2. Demo — the gate in action (~12 min)

When Vincent joins, show the full loop end to end, live:

- **Make a tiny real change on a branch** — say, tweak the footer text or a heading in `templates/index.html`. Branch, commit, push, open a PR.
- **Watch CI run on the PR** — the check spins, then goes green. Point out the **Merge** button is enabled *because* it's green.
- **Explain branch protection:** "If that check were red, GitHub wouldn't let me merge — and I can't push to `main` directly anymore either. Every change to the trusted version goes through this gate." Merge it, delete the branch, `git pull`.

You're demonstrating the thing this whole week built: not "I run tests," but "broken code *cannot* reach `main` — the repo enforces it."

## 3. The blocked merge — see it refuse (~10 min)

Prove the gate actually blocks, don't just claim it. On a branch, break a test on purpose (like W10D4 §6) *and* make a visible change, push, open a PR:

```
git switch -c demo-blocked
# break one assertion in tests/test_build_site.py
git add -A && git commit -m "Demo: break a test to show the gate block a merge"
git push -u origin demo-blocked
```

On the PR: CI goes red ✗, and now — because of this morning's branch protection — the **Merge button is disabled**, with "Required statuses must pass." *That's the difference from Thursday.* Thursday red was a warning you could ignore; today red is a locked door. Show Vincent the disabled button. Then throw the branch away:

```
git switch main
git push origin --delete demo-blocked
git branch -D demo-blocked
```

## 4. Break-and-catch — via CI this time (~10 min)

Vincent introduces a small break while you look away — but instead of you running `pytest` locally, the *robot* is the catcher. Vincent makes the change on a branch and opens a PR. Candidate breaks:

- A wrong `href` in `templates/index.html` (caught by `test_index_links_each_brand`).
- Deleting the `{% else %}` in `templates/brief.html` (caught by the fallback test).
- A subtler one: editing `db.all_brands` to drop the `ORDER BY` — does *any* test pin the ordering? (Maybe not — a gap worth noticing.)

Watch the PR's CI result.

**Caught it?** CI goes red, the merge is blocked, you read the failing test in the CI log, explain it, and the PR is closed. The gate did the catching — you didn't even have to run anything.

**Missed it?** CI goes *green* on a real break (like the dropped `ORDER BY` if nothing tests it). That's the important case: a green check is only as good as your tests. Name the gap, write the test that would have caught it, push it, watch CI now catch the break. This is the W5D5 lesson with teeth: CI doesn't make your tests *complete*, it makes them *unskippable*. A blind spot in your suite is a blind spot in your gate.

## 5. Claude judgment — extend the workflow (~10 min)

Open Claude Code:

```
claude
```

Ask:

> Update .github/workflows/ci.yml so it also runs the tests on Python 3.11 and 3.12 (a matrix), not just one version. Show the diff, don't apply.

Read the diff (W4D3 rules) and judge:

1. **Do you understand the `matrix` it added?** A build matrix runs the job once per listed version. If you can't read the YAML it produced, you can't accept it — that's the W4D1 rule (never accept a command/config you can't explain). Have Claude explain any line you don't follow *before* deciding.
2. **Is it in scope?** You asked for two Python versions. Did it also change the trigger events, add caching, or rename the job? Scope creep in a config file is as real as in code.
3. **Is it worth it here?** Brand Lens runs on your one Python version. Does testing two versions earn its complexity for *this* project, or is it cargo-culting a practice from bigger projects? A reasoned "reject — unnecessary for us" is a perfectly good verdict.

Decide accept / reject / modify and write the reason. (Either verdict is defensible — the skill is judging fit, not saying yes.)

## Looking ahead — onto the real internet

Your site runs on `localhost`, and now your workflow is professional: every change branches, gets reviewed, and passes CI before it touches `main`. There's one thing left that separates Brand Lens from a real product — *nobody but you can see it.* It lives on your laptop.

Week 11 puts it on the internet. You'll deploy the static site to **Amazon S3** — the same AWS that NIT ships to, so this is the real stack, not a toy. You'll make an AWS account, create a locked-down user for your laptop to use (not the root account — the same "don't use the master key for daily work" instinct as keeping secrets in `.env`), and learn the one command that copies your `site/` folder up to a bucket that serves it to the world. The foundations day covers what "the cloud" actually is, so it's a model and not a magic word.

And notice the order you built things in: tests (Weeks 5–9), then the CI gate (Week 10), *then* deploy (Week 11). That's deliberate. You only ever want *working, gated* code going out to the public — which in Week 12 you'll make literal: the deploy itself will run only when the tests are green.
