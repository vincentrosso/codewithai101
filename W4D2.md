## 2. Why this day exists (~5 min)

Yesterday you used Claude Code to summarize your repo. Today you go deeper: you have it explain the code you wrote, file by file, and you check.

Your code is a known quantity to you. That makes it the perfect test set for figuring out where Claude Code is reliable and where it isn't. If it's right about code you understand, you can lean on it for code you don't. If it's wrong about code you wrote, you have to stay skeptical even when you don't know the answer.

This is calibration. Same goal as W2D1's first Claude exercise; richer because Claude Code reads the actual files instead of a paste.

## 3. Walk through `runner.py` (~20 min)

In Claude Code:

> Read `runner.py` and explain it section by section. For each section, tell me what concern it owns and what it doesn't know about. Don't make any edits.

Read carefully. For each section Claude Code describes, check:

- **Accuracy.** Is the description of what the code does correct?
- **Confidence.** Is there a place where Claude Code states something with confidence that's actually a guess? (e.g. "this handles the case where..." — does it really, or is that an assumption?)
- **Gaps.** Did it miss anything important — a flag, a branch, an edge case?

If Claude Code says something that doesn't match your understanding, push back:

> You said `runner.py` does X. I thought it did Y. Which is right, and how do you know?

Watch how Claude Code responds when challenged. Sometimes it doubles down; sometimes it backs off; sometimes it admits uncertainty. Note the difference.

## 4. Walk through `fetch.py` (~15 min)

Same exercise, smaller file:

> Read `fetch.py` and explain it. What does it know about, and what does it deliberately not know about?

The "deliberately not know" framing is from W3D5's review prep — it forces Claude Code to talk about boundaries, not just behavior. See if it picks up on the modular split (fetch knows HTTP; parse knows HTML; runner knows orchestration). Or doesn't.

## 5. Walk through `parse.py` (~15 min)

Same exercise, the smallest of the three:

> Read `parse.py`. What does each function in here return, and in what shape?

This is the file most likely to have subtle bugs (HTML parsing has lots of edge cases). See what Claude Code surfaces. Common wins: it'll notice if a function returns `None` in some cases and a string in others. Common losses: it may invent edge cases that don't exist in your code.

## 6. The interesting question (~10 min)

Now ask something open-ended:

> Across `runner.py`, `fetch.py`, and `parse.py`, what's the most surprising decision the author made? Don't guess — point to a specific line.

Read what Claude Code surfaces. The interesting outcomes:

- **It points to something you remember struggling with.** Confirms your hunch.
- **It points to something you wrote without thinking.** That's a gap in your understanding — investigate the line until you can explain it.
- **It invents a "surprising" choice that isn't actually in the code.** It's filling the prompt with a plausible-sounding answer. Note this — it's an important failure mode to recognize.
