## 2. SELECT and WHERE — the two you'll use most (~15 min)

`SELECT` reads rows. Its shape:

```sql
SELECT <columns> FROM <table> WHERE <condition>;
```

Try each of these. **Predict the result before you run it** — jot the prediction in your journal:

```sql
SELECT url, title FROM pages;
SELECT url, title FROM pages WHERE brand_slug = 'innisfree';
SELECT url FROM pages WHERE status = 200;
SELECT url FROM pages WHERE status != 200;
SELECT url FROM pages WHERE error IS NOT NULL;
```

Three things to notice:

- **`SELECT *`** means "all columns." Naming the columns you want (`SELECT url, title`) is usually better — you get only what you need, and it keeps working if someone adds a column later.
- **`WHERE` filters rows.** Only rows where the condition is true come back. `=` tests equality (note: *one* equals sign in SQL, not two), `!=` is "not equal."
- **`IS NULL` / `IS NOT NULL`** — this is the gotcha. To test for missing data you do **not** write `WHERE description = NULL`. `NULL` means "unknown," and "unknown = unknown" is itself not true, so that query returns nothing. You must write `WHERE description IS NULL`. Try the wrong version and watch it return zero rows:

```sql
SELECT url FROM pages WHERE description = NULL;     -- returns nothing (wrong)
SELECT url FROM pages WHERE description IS NULL;    -- the right way
```

If your prediction for the first one was "rows with no description," edit your journal — that's the single most common SQL mistake and worth catching now.

## 3. COUNT — the question that motivated the week (~10 min)

`COUNT(*)` counts rows. Combine it with `WHERE` and you can answer the question from D1's intro in one line:

```sql

```

That's "how many pages have a meta description." No script, no loop. Compare it mentally to the JSON version (open file, parse, loop over brands, loop over pages, increment a counter) — same answer, one line.

## 4. ORDER BY (~5 min)

`ORDER BY` sorts the rows that come back:

```sql
SELECT url, status FROM pages ORDER BY status;
SELECT url, status FROM pages ORDER BY status DESC;
```

`DESC` is descending (high to low); the default is ascending. This is the SQL version of the `results.sort(...)` line you had in `runner.py` — except the database does the sorting and you just say what order you want.

## 5. GROUP BY — counting per category (~15 min)

This is the one that feels like a jump, so go slow. `GROUP BY` collapses rows that share a value into one row each, and lets you run `COUNT` (or other aggregates) within each group.

"How many pages does each brand have?"

```sql
SELECT brand_slug, COUNT(*) FROM pages GROUP BY brand_slug;
```

Read the output: one row per brand, with its page count. What happened mechanically: SQLite bucketed all the `pages` rows by `brand_slug`, then counted each bucket.

The mental model: **`WHERE` filters individual rows; `GROUP BY` + `COUNT` summarizes groups of them.** A query can do both:

```sql
SELECT brand_slug, COUNT(*)
FROM pages
WHERE status = 200
GROUP BY brand_slug;
```

"How many *successful* pages does each brand have." The `WHERE` thins the rows first; the `GROUP BY` buckets what's left.

**Predict, then run:** before you hit Enter on that last query, write down how many rows you expect back and what the counts should be, based on your seed data. Check. Edit your journal if you were off.

## 6. JOIN — using both tables at once (~15 min)

So far every query hit one table. The reason you have *two* tables is so a page can point at its brand without repeating the brand's full name in every page row. A `JOIN` stitches them back together when you want them together.

"Show each page's title alongside its brand's display name":

```sql
SELECT brands.name, pages.title
FROM pages
JOIN brands ON pages.brand_slug = brands.slug;
```

Read the `ON` clause as the instruction that makes the join meaningful: *match each `pages` row to the `brands` row where the page's `brand_slug` equals the brand's `slug`.* That `pages.brand_slug = brands.slug` link is exactly the foreign key you declared in the schema yesterday (`REFERENCES brands(slug)`), now being used.

You can join, filter, and group all at once — "how many successful pages per brand, by display name":

```sql
SELECT brands.name, COUNT(*)
FROM pages
JOIN brands ON pages.brand_slug = brands.slug
WHERE pages.status = 200
GROUP BY brands.name;
```

Don't worry about memorizing the full grammar. The shape to remember: `SELECT` columns `FROM` one table `JOIN` another `ON` how-they-match, then optionally `WHERE` / `GROUP BY` / `ORDER BY`. You'll look up the exact syntax for months; that's normal.

## 7. Queries from Python — and the `?` rule again (~15 min)

Running queries in the tool is for *you*. The pipeline runs them from Python. Add a query function to `db.py`:

```python
def pages_for_brand(conn, brand_slug):
    cursor = conn.execute(
        "SELECT url, title, status FROM pages WHERE brand_slug = ?",
        (brand_slug,),
    )
    return cursor.fetchall()
```

Run it interactively:

```
python3 -c "import db; conn = db.get_connection(); print(db.pages_for_brand(conn, 'innisfree'))"
```

You get a list of tuples, one per matching row. Three things:

- **`conn.execute(sql)` returns a cursor.** The cursor is the handle to the results. `.fetchall()` pulls every row as a list of tuples; `.fetchone()` pulls one.
- **The `?` placeholder, exactly as on D1.** `brand_slug` is a *value*, so it goes in as a parameter, never glued into the string. Notice `(brand_slug,)` — the trailing comma makes it a one-element tuple. `(brand_slug)` without the comma is just `brand_slug` in parentheses, not a tuple, and SQLite will reject it.
- **Why the `?` rule is non-negotiable, made concrete.** Imagine `brand_slug` came from user input and someone passed `x'; DROP TABLE pages; --`. With an f-string, that text becomes part of your SQL and could delete the table. With `?`, SQLite treats it as a literal string to match — it can't become commands. You're not exposed to that here, but building the reflex now means you never have to think about it later. **Values go in `?`; never f-string a value into SQL.**
