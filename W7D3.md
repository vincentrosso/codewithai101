## 2. An API is the W2D3 request/response, again (~10 min)

You already know this shape. In Week 2 (W2D3) you learned that `requests.get(url)` sends an HTTP **request** to a web server and gets back an HTTP **response** with a status code and a body. An API call is the *exact same thing*, pointed at a server whose job is to answer programs instead of browsers.

When you called `client.messages.create(...)` yesterday, under the hood the `anthropic` package:

1. Packaged your model name, messages, and `max_tokens` into an HTTP request.
2. Sent it over the internet to Anthropic's server (`api.anthropic.com`).
3. Attached your API key so the server knew who was asking.
4. Waited for the response, which came back as JSON.
5. Unpacked that JSON into the `message` object you read with `.content[0].text`.

The SDK hides steps 1–5 so you write Python, not raw HTTP. But it *is* HTTP. A web API (Application Programming Interface) is just "a website whose pages are meant for programs, not people" — you send structured data, you get structured data back. Same internet, same request/response, different audience.

This is why the word "API" shouldn't feel like new magic. It's the Week 2 model with a key attached and JSON on both ends.

## 3. The API key is authentication (~10 min)

A brand's website (Week 1) served you a page without asking who you were — it's public. Anthropic's API won't: every request must carry your **API key**, and the server checks it before doing anything. That check is **authentication** — proving you're you.

Three consequences worth internalizing:

- **The key is a password, not a username.** A username identifies you and is fine to share; a password authorizes actions and is not. The key authorizes spending money on your account. That's the entire reason it lives in `.env` and never in git (D1).
- **There's no "log out."** If a key leaks, anyone holding it can use it until you go to the console and *revoke* it. That's why a leaked key is an emergency: the fix is to delete the key and make a new one, immediately.
- **`Anthropic()` with no arguments still authenticates.** It's not skipping the key — it's reading it from the environment that `load_dotenv()` populated. The key is always sent; you just didn't type it in the code.

## 4. Tokens — the unit of everything (~10 min)

You set `max_tokens=300` without a full explanation. Here it is. A **token** is a chunk of text the model reads and writes in — very roughly **¾ of a word**, or about 4 characters of English. "Brand Lens evaluates Korean brands" is maybe 6–7 tokens.

Tokens are the unit of three things at once:

- **Cost.** You're billed per token — so many fractions of a cent per thousand tokens *in* (your prompt) and per thousand *out* (Claude's reply). Output usually costs more than input. This is why a short prompt and a `max_tokens` cap keep the bill near zero, and why dumping a giant document into every call adds up.
- **Limits.** `max_tokens` caps the *output* — the most Claude will generate before it stops. It's a safety valve against a runaway (and expensive) response. There's also a separate, much larger limit on *total* tokens per call (the "context window") — how much prompt + reply can fit at once.
- **Rate limits.** Separately from per-call size, the API limits how many tokens (and requests) you can send *per minute*. Go over and the server replies with an error (HTTP 429, "too many requests") instead of an answer. A program that calls the API in a loop has to expect this — which is part of why D4 wraps the call in error handling.

So when D2 said "this costs fractions of a cent," tokens are the meter. A small model (Haiku) and a `max_tokens` cap are you keeping the meter low.

## 5. What a model is, and what it isn't (~10 min)

This is the most important section, because it changes how much you trust the output.

Claude is a **text predictor**. Given the text so far (your prompt), it predicts plausible next text, token by token. It was trained on an enormous amount of writing, so its predictions are often genuinely useful — but "plausible" is not "true." Three properties follow, and you saw all three this week:

- **It's non-deterministic.** Same prompt, different answer each time (D1, D2). Internally the model has a list of likely next tokens and picks among them with some randomness — a setting called *temperature*. Higher temperature, more varied; lower, more repetitive. The default isn't zero, so you get variety. A pure function (`parse_page`) has no such knob; an LLM always does.
- **It can be confidently wrong — "hallucinate."** Because it predicts *plausible* text, it can produce a fluent, confident statement that is simply false: a value proposition for a brand that the scraped data never supported, a fact about a company that isn't true. It doesn't "know" it's wrong, because it isn't checking against truth — it's predicting text. This is why D2's prompt asked for a `confidence` field and tied each claim to a specific scraped field: you're trying to keep the model *grounded in the data you gave it* rather than free-associating.
- **It only sees what you send it.** The model has no access to your database, your files, or today's news unless you put it in the prompt. The brand summary is built *only* from the fields you passed. If `description` was `None`, the model is guessing from less — and a good prompt makes it say so (low confidence) rather than inventing.

The practical upshot for Brand Lens: the LLM is a powerful *drafter of judgments*, not an oracle. D5 will make you read its output and ask, every time, "is this grounded in the data, or did it make it up?"

## 6. Why you can't unit-test a real LLM call (~10 min)

Pull the Week 5 model forward. A good unit test is fast, deterministic, and offline (W5D3). A real call to `client.messages.create` is the opposite of all three:

- **Not fast** — a network round-trip plus model generation, often a second or more. A suite of these would be too slow to run constantly.
- **Not deterministic** — the whole of section 5. You literally cannot write `assert result["value_proposition"] == "..."` because the wording changes every run. The assertion would be flaky by construction (W5D3 §7).
- **Not offline, and not free** — it needs the internet, a valid key, and it costs money *every time the test runs*. Tests run hundreds of times. You don't want each run billing your account.

This is the same situation as `fetch_one` in W5D4: a function with an unavoidable, slow, unreliable side effect. And the answer is the same — **mock it.** In the test, you replace `client.messages.create` with a fake that returns a canned response, instantly, offline, for free. Then you test *your* logic — does `summarize_brand` build the prompt right, dig out `.content[0].text`, and parse the JSON? — without testing Claude itself (which isn't your job to test). That's exactly what you'll build tomorrow. Today, just hold the reasoning: **you mock the LLM for the same three reasons you mocked the network.**
