## About this week — why testing, why now

You spent Week 4 letting Claude Code into the repo. Every edit it makes lands in your working tree, and your only defense is reviewing the diff (W4D3). That defense has a hole: a diff can *look* fine and still break something subtle. You read every `+` line, accepted the change, ran the script once, it printed something reasonable, you committed. Two days later you notice the canonical URL is always `None`. When did that break? You don't know.

Tests close the hole. A test is a small program that calls one of your functions with known input and checks the output is what you expected. You write the test once. Every time you run `pytest`, it re-runs every check. If a future change — yours or Claude Code's — breaks something a test covered, the test fails and tells you exactly which case broke. The earlier the alarm, the cheaper the fix.

This week:

- **Today (D1):** install pytest, first test against `parse_page`, the AAA shape.
- **D2:** more tests, edge cases, fixtures for shared setup.
- **D3:** foundations day — no code, no AI, the mental model behind testing.
- **D4:** test `fetch.py` without hitting the live internet (mocking).
- **D5:** Friday review with Vincent — Vincent breaks one of your functions, the tests should catch it.

## 2. Install pytest (~5 min)

`pytest` is a third-party library (like `requests` and `beautifulsoup4` from Week 1). Install it into your venv:

```
pip install pytest
```

Verify:

```
pytest --version
```

Should print a version number like `pytest 8.x.x`. If you see `command not found`, your venv isn't activated — run `source .venv/bin/activate` and try again.

If your project has a `requirements.txt`, add a line for it:

```
pytest>=8.0
```

If it doesn't yet, don't worry about it today. We'll deal with `requirements.txt` properly later.

## 3. Make a tests/ directory (~5 min)

```
mkdir tests
touch tests/__init__.py
```

Two things to notice:

- **`tests/` lives next to your code**, not inside it. `fetch.py`, `parse.py`, `runner.py`, `brands.yaml` stay at the root; tests get their own folder. The convention is so universal that pytest finds the directory automatically.
- **`__init__.py` is an empty file** that marks `tests/` as a Python package. It lets your test files import from your project (e.g., `from parse import parse_page`) without import shenanigans.

Confirm the structure:

```
ls
ls tests
```

You should see `tests/` at the root and `__init__.py` inside it.

## 4. Write your first test (~20 min)

```
touch tests/test_parse.py
subl tests/test_parse.py
```

```python
from parse import parse_page


def test_parse_page_extracts_title():
    html = """
    <html>
      <head>
        <title>Hello World</title>
      </head>
      <body></body>
    </html>
    """
    record = parse_page(html, "https://example.com/")
    assert record["title"] == "Hello World"
```

Read every line. There are only three ideas, but they're each important:

- **`from parse import parse_page`** — same import you'd write in `runner.py`. The test file reaches into your code the same way any other module would.
- **`def test_parse_page_extracts_title():`** — a regular Python function. The name *must* start with `test_` — that's the convention pytest uses to discover tests. The rest of the name describes what's being verified, in plain language.
- **`assert record["title"] == "Hello World"`** — the actual check. `assert` is a built-in Python keyword: if the expression after it is true, nothing happens; if it's false, the test fails. A test with no `assert` is not a test — it's a function that runs and proves nothing.

The structure of the test body follows a pattern people call **AAA** — Arrange, Act, Assert:

1. **Arrange.** Build the inputs. The `html` string is hand-written so you know exactly what's in it.
2. **Act.** Call the function under test. `parse_page(html, url)` is the only line that does real work.
3. **Assert.** Check the output matches your expectation. One assertion is fine; many are fine. The point is to compare *actual* against *expected*.

You'll see this shape over and over. When you read someone else's tests, look for the three blocks.

## 5. Run pytest (~5 min)

From the project root (`~/dev/brand-lens`, *not* inside `tests/`):

```
pytest
```

You should see something like:

```
============================= test session starts ==============================
collected 1 item

tests/test_parse.py .                                                    [100%]

============================== 1 passed in 0.04s ===============================
```

The dot after `test_parse.py` means *one test passed*. A failing test would show an `F`. An error during collection (e.g., import error) shows an `E`.

For more detail, use `-v` (verbose):

```
pytest -v
```

Now you see the test name spelled out. `-v` is the flag you'll want most days.

## 6. Break it on purpose (~10 min)

A passing test only proves something if you've seen it fail at least once. Otherwise you don't know whether the assertion is actually checking what you think.

Open `parse.py`. Find the line that sets `record["title"]`. Change it to set the wrong value — for example:

```python
if soup.title and soup.title.string:
    record["title"] = "WRONG"
```

Run pytest again:

```
pytest -v
```

You should see a failure like:

```
FAILED tests/test_parse.py::test_parse_page_extracts_title - AssertionError: assert 'WRONG' == 'Hello World'
```

Read it. pytest tells you:

- *Which test failed* — `test_parse_page_extracts_title`
- *What the assertion was* — `assert 'WRONG' == 'Hello World'`
- *Why it failed* — the left side ("actual") didn't match the right side ("expected")

That's the loop: the test caught the regression *the moment you ran it*. Without it, you might've shipped the broken parser and not noticed until a brand's title looked wrong in tomorrow's output.

Now fix `parse.py` — put the original line back:

```python
if soup.title and soup.title.string:
    record["title"] = soup.title.string.strip()
```

Run pytest one more time:

```
pytest
```

Green again. You just did the full cycle: green → red → green. That cycle is the entire heartbeat of testing.

## 7. The shape of a test, one more time (~5 min)

Look at `tests/test_parse.py`. Notice what a test isn't:

- A test does **not** `print` and then have you eyeball the output. The `assert` does the checking.
- A test does **not** `return` a value. pytest ignores return values — only assertions count.
- A test does **not** call your function with live network input. The `html` string was hand-crafted so you know what's in it. (Tomorrow you'll lean on this hard: every parse test will use hand-written HTML.)
- A test file is **not** run with `python3 tests/test_parse.py`. You run `pytest` from the project root, and pytest does the discovery and reporting.

That's it. Tests are small functions named `test_*` that assert something about your code.
