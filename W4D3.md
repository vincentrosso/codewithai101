## 2. What a diff is (~10 min)

A **diff** shows the difference between two versions of a file: what was there before, what's there now, what changed.

Imagine `runner.py` line 42 used to say:

```python
limit = 10
```

And now it says:

```python
limit = args.limit or 10
```

A diff represents that as:

```
- limit = 10
+ limit = args.limit or 10
```

The `-` line is the old version (deleted). The `+` line is the new version (added). Lines with neither prefix are **context** — unchanged lines around the change, shown so you can see where the change sits.

A diff isn't just for git. It's any tool's way of saying "here's the change." Claude Code shows diffs the same way before applying edits.

## 3. Anatomy of a `git diff` output (~15 min)

Open your terminal in `~/dev/brand-lens`. Make a small change to `runner.py` — change one line, doesn't matter what. Don't run `git add`. Run:

```
git diff
```

You'll see something like:

```
diff --git a/runner.py b/runner.py
index 1a2b3c4..5d6e7f8 100644
--- a/runner.py
+++ b/runner.py
@@ -40,7 +40,7 @@ def main():
     args = parser.parse_args()
     with open(args.input) as f:
         config = yaml.safe_load(f)
-    limit = 10
+    limit = args.limit or 10
     for slug, brand in config["brands"].items():
         pages = []
```

Read top to bottom:

- **`diff --git a/runner.py b/runner.py`** — header. `a/` is the old version, `b/` is the new.
- **`index 1a2b3c4..5d6e7f8 100644`** — git's internal hashes for the old and new content. Ignore for now.
- **`--- a/runner.py`** and **`+++ b/runner.py`** — naming the old and new files explicitly.
- **`@@ -40,7 +40,7 @@ def main():`** — a **hunk header**. Reading it: "starting at line 40 of the old file, 7 lines; starting at line 40 of the new file, 7 lines." The `def main():` is the nearest enclosing function — git shows it for context.
- **The hunk body** — the actual change. Lines starting with `-` are removed, `+` are added, no prefix means unchanged context.

That's the whole format. Most of what you'll read is hunk headers and `+`/`-` lines.

## 4. Practice — make changes and read diffs (~15 min)

Still in `~/dev/brand-lens`. Run each of these and read the output before moving on:

```
git diff
git diff runner.py
git diff --stat
git diff --color-words
```

- `git diff` (no args) — all unstaged changes in the working tree.
- `git diff runner.py` — only changes to that file.
- `git diff --stat` — summary: how many lines changed in which files. Useful when the diff is huge.
- `git diff --color-words` — highlights the changed *words*, not just lines. Easier to spot a one-character change in a long line.

Now revert your change so the working tree is clean:

```
git checkout runner.py
```

(That throws away unstaged changes to that file. Safe today because you didn't intend to keep the change. Don't run this casually outside of practice — it's destructive.)

## 5. Staged vs unstaged (~10 min)

There are three places code can live:

1. **Working tree** — the files on disk you're editing.
2. **Staging area (index)** — what `git add` puts files into; what your next commit will include.
3. **Last commit** — the most recent committed state.

`git diff` shows working tree vs. staging area — what you've changed but haven't staged.

`git diff --cached` (or `git diff --staged`) shows staging area vs. last commit — what you've staged but haven't committed.

`git diff HEAD` shows working tree vs. last commit — everything that's changed since the last commit, staged or not.

Why this matters: when Claude Code edits a file, the change goes into the working tree. If you `git add` it, it moves to staging. If you `git commit`, it lands in the last commit. Each move is a chance to review with the right `git diff` flag. Tomorrow you'll use `git diff` and `git diff --cached` together.

## 6. Spotting a bad diff (~10 min)

Not every diff is what it claims to be. Things to look for when reviewing a Claude Code edit:

- **Scope creep.** You asked for a `--brand` flag. The diff also reformats unrelated lines, "fixes" a comment, or adds an import you didn't ask about. Each of those changes is a place a bug can hide.
- **Deletions you didn't ask for.** A `-` line you didn't expect is the highest-risk part of any diff. Claude Code may have decided a line was unnecessary. It may have been wrong.
- **Renames disguised as edits.** If a function gets renamed, you'll see a `-` for the old name and a `+` for the new. But every *call site* needs to change too. If the diff renames a function but doesn't update every caller, the code will break.
- **"Improvements" you didn't ask about.** Claude Code sometimes throws in a small refactor or stylistic change. Even if it's correct, it's outside scope. Reject it and ask for the original change only.
- **A diff much bigger than you expected.** If you asked for a five-line change and got a 50-line diff, something is off. Ask Claude Code why before reading further.

The general rule: **every line in a diff has to earn its place.** If you can't say why a `+` or `-` line is there, don't accept it.
