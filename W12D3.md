## 2. The two journeys (~20 min)

There are two completely different paths through Brand Lens, and the heart of understanding the system is seeing that they're *separate*. Confusing them is the single most common beginner error about web apps — and you, having built a static site, get to see exactly why they're separate.

**Journey 1 — building the site (happens ahead of time, by you / the robot).** This is the pipeline, the thing twelve weeks built:

```
brands.yaml                       (W3: which brands, which pages)
    │
    ▼  runner.py orchestrates:
fetch.py   ── get each page off the network        (W1–2)
    ▼
parse.py   ── HTML → fields (title, desc, og, h1, canonical)  (W2D4)
    ▼
db.py      ── store pages in SQLite                (W6)
    ▼
brief.py   ── ask Claude → structured JSON brief   (W7)
    ▼  (stored back in the db)                      (W7D4)
build_site.py ── db → render HTML → site/          (W9)
    ▼
deploy.sh / the robot ── sync site/ → S3 bucket    (W11–12)
```

Every arrow is something you wrote. This journey runs when the pipeline runs — on your laptop, or on the robot on a merge. By the time it finishes, the site is finished files in a bucket.

**Journey 2 — a visitor viewing the site (happens later, per click).** This is almost nothing, *on purpose*:

```
visitor's browser  ── "GET index.html" ──▶  S3 bucket
visitor's browser  ◀── the file ──────────  S3 bucket
browser renders the HTML/CSS into a page
```

That's the whole of it. The visitor's request runs *no code of yours*. There's no `fetch.py`, no LLM, no database in Journey 2 — S3 just hands over a file that was already built in Journey 1. **That emptiness is what "static" means** (W9D3), and it's why the deploy was so simple: there's no running program to host, only files to hand out.

The big idea: **all the work is in Journey 1, none in Journey 2.** A *dynamic* site would move work into Journey 2 — running a backend on every visit. You deliberately kept Journey 2 empty, and bought yourself the simplest possible deploy in exchange.

## 3. What "deployed" actually means (~10 min)

You've said "deploy" all week; pin it down. **To deploy is to move your built software from where you make it (your laptop) to where it runs/serves for real (the public internet).** For Brand Lens, deploying = copying `site/` into the S3 bucket that the public can read. Before deploy, the site exists only where you can see it; after, it exists where anyone can.

Notice "deployed" is *not* a property of your code — it's a property of *where a copy of it lives*. The same `index.html` is "not deployed" on your disk and "deployed" in the bucket. That's why deploy is a *step* (a copy), not a code change.

## 4. What "CI/CD" actually means, now that you've lived it (~15 min)

Two robots, two jobs, one pipeline:

- **CI — continuous integration:** every time code arrives, automatically integrate and *test* it. Your `test` job running `pytest` on every push and PR (Week 10). It answers: *does this change keep everything green?*
- **CD — continuous deployment:** every time the trusted branch updates *and the tests pass*, automatically *deploy* it. Your `deploy` job, `needs: test`, on `main` (Week 12). It answers: *get the green change to the public, with no hand-cranking.*

The shape you built — **propose (branch) → review (PR) → test (CI) → merge → deploy (CD)** — is the actual loop professional teams run, end to end. And the *order you learned it in was the point*: tests first (5–9), then a gate (10), then deploy (11), then automation (12). You never deploy ungated code, because the gate existed before the deploy did. The discipline is baked into the sequence.

## 5. The full map — twelve weeks (~10 min)

Read this once, slowly. Every line is something you can now do.

- **W1** — terminal, Python, git: the ground floor (AI-free, on purpose).
- **W2** — functions, dicts, HTTP, parsing fields. Claude as a tutor.
- **W3** — config (`brands.yaml`), modules (`fetch`/`parse`/`runner`), a CLI.
- **W4** — Claude Code: review every diff, never accept what you can't explain.
- **W5** — tests: pytest, fixtures, mocking the network.
- **W6** — a real database: SQLite, schema, SQL, upsert.
- **W7** — the LLM as a library: API keys, structured output, mock it in tests.
- **W8** — the brief MVP: templates, one document per brand.
- **W9** — a static website: HTML/CSS, build the whole site from the db.
- **W10** — branches, PRs, CI: the test gate.
- **W11** — AWS S3: the site on the real internet.
- **W12** — CI/CD: it tests and ships itself.

From "what's a terminal" to "a tested, gated, self-deploying web tool." That's the arc.
