## 2. Why `fetch.py` is hard to test (~10 min)

Re-read `fetch.py`:

```python
import requests


def fetch_one(url):
    """Fetch a URL and return (status, html, error). Never raises."""
    try:
        response = requests.get(url, timeout=10)
        return response.status_code, response.text, None
    except Exception as e:
        return None, None, str(e)
```

If you wrote a test that called `fetch_one("https://example.com")` directly, three things would go wrong:

1. **The test is slow.** Network round-trips take hundreds of milliseconds. A suite with 50 such tests takes most of a minute. You stop running them.
2. **The test is flaky.** `example.com` could be down. Your wifi could blip. A DNS lookup could time out. The test fails for reasons unrelated to your code. (See W5D3 §7.)
3. **The error path is impossible to trigger reliably.** How do you make `requests.get` *raise* on demand to test that the `except` branch works? Pull your ethernet cable mid-test? You can't.

The clean answer is to **not actually call `requests.get` during the test.** Substitute a fake version that returns whatever you want — including a fake that raises. That technique is called **mocking**, and it's the whole subject today.

## 3. The idea of a fake (~5 min)

A **mock** (or **fake**, or **stub** — the words overlap; nobody is strict) is something your test puts in place of the real thing, just for the duration of that test. It has the same shape as the real thing — same methods, same attributes — but it returns whatever you tell it to.

For `fetch_one`, the real thing is `requests.get`. The fake will be a Python function you write inside the test that mimics what `requests.get` would do, but without touching the network.

You need two things to pull this off:

1. A way to substitute the fake in place of the real `requests.get` during the test, and put the real one back afterward. That's `monkeypatch`.
2. A fake response object that has the attributes `fetch_one` reads — `status_code` and `text`.

## 4. monkeypatch — pytest's built-in (~15 min)

`monkeypatch` is a fixture pytest provides for free. You ask for it the same way you asked for `full_page_html` on D2 — by adding it as a parameter:

```python
def test_something(monkeypatch):
    ...
```

The fixture gives you an object with one method you care about: `setattr(target, name, value)`. It replaces `target.name` with `value` for the duration of the test, then puts the original back automatically when the test ends. Other tests are unaffected.

For our case, we want to replace `requests.get`. So inside a test we'll write:

```python
monkeypatch.setattr(requests, "get", fake_get)
```

Where `fake_get` is a function we wrote.

The reason `monkeypatch` exists as a fixture (instead of just doing `requests.get = fake_get` directly) is **isolation**. If a test does the assignment manually and crashes before restoring, every subsequent test in the run sees the broken `requests.get`. `monkeypatch` guarantees cleanup.

## 5. Test the happy path (~15 min)

Make a new test file:

```
touch tests/test_fetch.py
subl tests/test_fetch.py
```

```python
import requests

from fetch import fetch_one


class FakeResponse:
    def __init__(self, status_code, text):
        self.status_code = status_code
        self.text = text


def test_fetch_one_returns_status_and_html_on_success(monkeypatch):
    def fake_get(url, timeout):
        return FakeResponse(200, "<html><body>hello</body></html>")

    monkeypatch.setattr(requests, "get", fake_get)

    status, html, error = fetch_one("https://example.com/")

    assert status == 200
    assert html == "<html><body>hello</body></html>"
    assert error is None
```

Read every part:

- **`class FakeResponse`** — the fake response object. It only has the two attributes `fetch_one` actually reads: `status_code` and `text`. It doesn't need anything else, because nothing else gets touched. Fakes are minimum-viable on purpose.
- **`def fake_get(url, timeout)`** — the fake function that will stand in for `requests.get`. Its signature has to match what `fetch_one` calls (`requests.get(url, timeout=10)`), but the body can do whatever you want. Here it ignores `url` and `timeout` and just returns a canned response.
- **`monkeypatch.setattr(requests, "get", fake_get)`** — the swap. After this line, anywhere in `fetch.py` that calls `requests.get(...)` will land in `fake_get` instead.
- **`status, html, error = fetch_one(...)`** — Act. Looks identical to a real call. `fetch_one` doesn't know or care that `requests.get` is a fake today.
- **Three asserts** — verify the tuple `fetch_one` returns. Three small statements, each protecting one promise.

Run it:

```
pytest -v
```

Eight tests passing — seven for parse, one for fetch.

## 6. Test the error path (~15 min)

Now the test that wouldn't have been possible without mocking. Add to `tests/test_fetch.py`:

```python
def test_fetch_one_returns_error_when_request_raises(monkeypatch):
    def fake_get(url, timeout):
        raise requests.RequestException("network is down")

    monkeypatch.setattr(requests, "get", fake_get)

    status, html, error = fetch_one("https://example.com/")

    assert status is None
    assert html is None
    assert error == "network is down"
```

What just happened:

- The fake doesn't return a response. It raises a `requests.RequestException`, simulating what a real network failure would look like.
- Inside `fetch_one`, the `try` block hits that exception. The `except Exception as e` catches it. The function returns `(None, None, "network is down")` — the error tuple.
- The test asserts on that tuple. If anyone ever removes the `try/except` from `fetch.py`, this test fails immediately and tells you exactly what's broken.

Run it:

```
pytest -v
```

Nine tests passing. Pause and notice what just happened: **you wrote a test for code that handles a situation you can't reliably produce in real life.** This is the superpower of mocking. The error branch in `fetch_one` was speculative when you wrote it on W1D4 — "hopefully it does the right thing." Now it's verified.

## 7. Optional — test a non-200 status (~10 min)

If you have time, add one more:

```python
def test_fetch_one_passes_through_non_200_status(monkeypatch):
    def fake_get(url, timeout):
        return FakeResponse(404, "<html><body>not found</body></html>")

    monkeypatch.setattr(requests, "get", fake_get)

    status, html, error = fetch_one("https://example.com/missing")

    assert status == 404
    assert html == "<html><body>not found</body></html>"
    assert error is None
```

This test documents an interesting choice your code makes: a 404 isn't treated as an error — it's just a non-200 status with body content, and the *caller* decides what to do with it. That's a real design decision, and now there's a test that names it. If someone later changes `fetch_one` to raise on non-2xx, this test fails and forces the conversation.
