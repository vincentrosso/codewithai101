# Week 10, Day 4 — Continuous Integration: A Robot Runs Your Tests
*v1.0.0*

**Goal for today:** set up **continuous integration** (CI) — a robot on GitHub that runs your entire pytest suite automatically on every push and every pull request. You'll write the workflow file, watch it run green, then deliberately break a test on a branch and watch the PR turn red. By end of day your tests run themselves, on someone else's computer, every time you propose a change.

**Time budget:** ~80 min hands-on, ~15 min journal, ~15 min slack.

---

## 1. Warm-up (~5 min)

```
source .venv/bin/activate
git switch main
git pull
pytest -v
```

On `main`, synced, green. Count your tests — you'll watch this exact number run on GitHub in a few minutes.

## 2. What CI is, and why it's the week's payoff (~10 min)

You run `pytest` by hand before pushing — *when you remember*. **Continuous integration** removes the "when you remember." It's a robot that watches your repo and, every time code arrives (a push, a PR), automatically: checks out the code, installs the dependencies, and runs your tests — on a fresh machine in the cloud, not your laptop. The result shows up as a green check ✓ or a red ✗ right on the commit and the PR.

This is the literal cash-in of Weeks 5–9. You wrote tests; the [why-test](sidelines/why-test.md) sideline's reason #2 was "tests are how you trust a change." CI makes that trust *automatic and unskippable*: a change that breaks the suite gets a red ✗ that everyone can see, before it merges. The tests stop being a chore you might skip and become a wall broken code runs into.

The tool you'll use is **GitHub Actions** — CI built into GitHub, free for your purposes, configured by a single file in your repo.

## 3. Why your tests can even run on a robot (~10 min)

Here's a question worth pausing on, because the answer is something you *built* without knowing you'd need it today. The CI robot is a bare machine in a data center. It has **no internet access to your target sites, no `ANTHROPIC_API_KEY`, no `.env` file** (that's gitignored — it never left your laptop). So how can it run tests for `fetch_one` (which hits the network) and `summarize_brand` (which calls a paid API)?

Because **you mocked them.** In W5D4 you replaced `requests.get` with a fake. In W7D4 you replaced the Anthropic client with a fake. Your tests for the network and the LLM *never actually touch the network or the LLM* — they run against fakes, offline, instantly, for free. That design choice, made weeks ago for speed and reliability, is exactly what lets a secretless robot with no internet run your whole suite. If your tests had needed a real key or a live site, CI would be a nightmare of leaked secrets and flaky network calls. They don't. (This is the foundations-day point from W7D3 — "you can't unit-test a real API call, so you mock it" — paying a dividend you couldn't have predicted.)

So the robot needs nothing but your code and `requirements.txt`. That's the whole reason this is easy.

## 4. Write the workflow file (~20 min)

GitHub Actions looks for workflow files in a specific directory: `.github/workflows/`. Create it and a `ci.yml`:

```
mkdir -p .github/workflows
subl .github/workflows/ci.yml
```

```yaml
name: CI

on:
  push:
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Check out the code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run tests
        run: pytest -v
```

This is **YAML** — a config format that's just labels, indentation, and lists (you met it in W3D1 with `brands.yaml`). Read it top to bottom:

- **`name: CI`** — what this workflow is called in the GitHub UI.
- **`on:`** — *when* to run. `push` and `pull_request` mean "every time code is pushed, and every time a PR is opened or updated." These are events on the commit graph you learned yesterday.
- **`jobs:` → `test:`** — one job, named `test`. A workflow can have many jobs; you need one.
- **`runs-on: ubuntu-latest`** — the robot's operating system: a fresh Linux machine GitHub spins up for you, uses, and throws away.
- **`steps:`** — the job, in order. Two kinds of step:
  - **`uses:`** — run a pre-made action someone published. `actions/checkout@v4` copies your repo onto the robot; `actions/setup-python@v5` installs Python. You don't write these; you *use* them.
  - **`run:`** — run a shell command, exactly what you'd type. `pip install -r requirements.txt` (the recipe from W8D4 earning its keep — the robot installs your *pinned* dependencies) and `pytest -v` (the suite).

The logic is just your warm-up routine — check out, install, test — written so a robot does it. Nothing here you haven't done by hand a hundred times.

## 5. Ship it through a PR (~15 min)

Set up the CI *using the workflow you just learned* — branch, PR, merge. (A little recursive: you're using the branch/PR flow to add the robot that guards the branch/PR flow.)

```
git switch -c add-ci
git add .github/workflows/ci.yml
git commit -m "Add GitHub Actions CI to run pytest on every push and PR"
git push -u origin add-ci
```

Open the PR on GitHub. Now watch something new: at the bottom of the PR, a **check** appears — "CI / test" — with a spinner. GitHub noticed the push, read your `ci.yml`, spun up a Linux machine, and is running your suite. Click **Details** to watch the live log: checkout, Python setup, `pip install`, then your tests scrolling by. After a minute or two: a green ✓ **"All checks have passed."**

Your tests just ran on a computer that isn't yours, triggered by nothing but a push. Merge the PR, delete the branch, sync:

```
git switch main
git pull
git branch -d add-ci
```

## 6. Make it go red — on purpose (~15 min)

A green check you've never seen turn red is a smoke detector you've never tested. Break a test on a branch and watch CI catch it — *before* it could reach `main`.

```
git switch -c break-on-purpose
```

Open `tests/test_build_site.py` and sabotage one assertion — change `test_index_links_each_brand` to expect a wrong link:

```python
    assert 'href="WRONG.html"' in html   # deliberately wrong
```

Commit and push:

```
git add tests/test_build_site.py
git commit -m "Temporarily break a test to watch CI catch it"
git push -u origin break-on-purpose
```

Open the PR. This time the check spins, then turns **red ✗** — "Some checks were not successful." Click **Details**: the log shows `pytest` failing on exactly that assertion, with the same red output you'd see locally — except it ran itself, on the PR, where everyone can see it. *This is the gate working.* A broken change announced itself, loudly, before merge.

Now undo it — **close the PR without merging** and delete the branch:

```
git switch main
git push origin --delete break-on-purpose      # delete the remote branch
git branch -D break-on-purpose                  # delete the local branch (capital -D: discard unmerged)
```

(Close the PR on GitHub too if it's still open.) You never want that break on `main`; the point was only to see the robot do its job. Confirm `main` is green and clean.
