## 4. The grand break-and-catch (~12 min)

Vincent breaks something — but this time it could be *anywhere* in the system, and the skill is knowing *which layer* and *what would catch it*. Candidates span the whole stack:

- A code break (wrong `href`, deleted `{% else %}`) → **CI catches it**, the deploy is skipped, nothing ships.
- An infra break (re-block public access, break the bucket policy) → **CI stays green, the live site 403s** — only your eyes catch it (W11D5).
- A pipeline break (a parse field returns wrong) → maybe a test catches it, maybe a *grounding* check does.
- A deploy break (the `deploy.yml` syncs the wrong folder) → the Actions log catches it.

For whatever Vincent does, the question is the same three-part diagnosis: **which layer broke (code / build / infra / deploy)? what's the first place you'd look? what *should* have caught it, and did it?** This is the synthesis of every Friday: by now you have *multiple* catchers — the test gate, your eyes on the live site, the grounding check, the Actions log — and the mastery is knowing which one owns which failure.

**Caught it?** Diagnose the layer, fix it, confirm green-and-live.
**A gap?** Name it precisely — and notice whether it's a *test* gap (write the test) or a *category CI can't cover* (infra), which is a real and fair answer, not a failure.

## 5. Claude judgment — the finished system (~10 min, optional if short on time)

One last Claude Code pass, aimed at the *whole* tool now:

```
claude
```

Ask:

> Look at this project end to end — the pipeline, the static site, the CI/CD workflow. Suggest the single highest-value improvement you'd make next, and why. Don't write code; make the case.

Read its answer with everything you've learned about judging Claude:

1. **Is it grounded in *your* project**, or generic best-practice boilerplate true of any repo? (The W7 grounding instinct, pointed at advice instead of a brief.)
2. **Do you agree with the priority?** You know this system better than Claude does now. Is its "highest-value" actually highest-value for Brand Lens, or does it miss something you'd rank above it?
3. **Could you scope and review the change** if you decided to do it? That you *can* now judge whether to take an AI's architectural advice — rather than just accepting it — is the whole point of how this course taught you to work with AI.

Write your verdict and your *own* pick for what you'd do next. (Hold that thought — section 6 asks for it, and it's the seed of whatever you build after this.)
