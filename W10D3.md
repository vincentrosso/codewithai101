## 2. A commit is a snapshot with a parent (~15 min)

You've made dozens of commits. Here's what one *is*: a **snapshot** of your whole project at a moment, plus a pointer to the commit that came *before* it (its **parent**), plus a message and an author.

Because each commit points to its parent, the commits form a **chain** — a links a back through history. Commit C points to B, B points to A, A is the start. Follow the parent pointers backward and you can reconstruct the entire history. This chain (really a *graph*, once branches enter) is what git stores. Everything else — branches, merges — is built on top of "commits pointing at their parents."

Draw it for yourself (this is the picture the rest of the day uses):

```
A ─── B ─── C
```

Three commits, each pointing back to the one before. `C` is the latest.

## 3. A branch is a pointer, not a copy (~20 min)

Here's the idea that surprises everyone, so sit with it: **a branch is not a copy of your files. A branch is a tiny label that points at one commit.** That's all. `main` is a label pointing at a commit. When you make a new commit on `main`, the *label* slides forward to the new commit.

Watch it move. Start here — `main` points at `C`:

```
A ─── B ─── C  ← main
```

You run `git switch -c add-site-footer`. This creates a *second label* pointing at the same commit, `C`, and points `HEAD` (a special marker meaning "the branch you're currently on") at it:

```
A ─── B ─── C  ← main, add-site-footer (HEAD)
```

No files copied — both labels point at the same commit `C`. Now you commit the footer. The new commit `D` has `C` as its parent, and the label you're *on* (`add-site-footer`) slides to `D`. `main` stays put:

```
A ─── B ─── C  ← main
             \
              D  ← add-site-footer (HEAD)
```

*This is the picture* behind "main doesn't have that commit" from W10D1. `main` literally still points at `C`; your footer is commit `D`, reachable only from the `add-site-footer` label. The two labels have diverged. Nothing was copied or duplicated — git just has two pointers into one graph.

Why this matters: because a branch is just a cheap pointer, making one is instant and free, and you can have as many as you like. That's *why* "one branch per change, throw it away after" is practical advice and not wasteful — a branch costs almost nothing. Switching branches just moves `HEAD` and updates your working files to match that commit.

## 4. A merge joins two lines back together (~15 min)

When you merged the PR, git brought `D` into `main`. In the simplest case — `main` hadn't moved since you branched — git can just slide the `main` label forward to `D`:

```
A ─── B ─── C ─── D  ← main, (add-site-footer deleted)
```

That's called a **fast-forward** merge: nothing to reconcile, just move the label. Clean and common for small solo changes.

But suppose, while you worked on `D`, `main` had *also* gained a commit `E` (a teammate, or you in another branch):

```
A ─── B ─── C ─── E   ← main
             \
              D       ← add-site-footer
```

Now git can't just slide a label — both lines have new work. It creates a **merge commit** `M` with *two* parents (`E` and `D`), tying the histories together:

```
A ─── B ─── C ─── E ─── M  ← main
             \         /
              D ───────
```

`M` says "main as it was, combined with the footer branch." Most of the time git figures out how to combine the two sets of changes automatically. When both lines edited *the same lines of the same file* in conflicting ways, git can't guess which to keep — that's a **merge conflict**, and git stops and asks *you* to choose. You haven't hit one yet (solo work, one branch at a time); just know the word and that a conflict is git refusing to silently pick — a safety feature, not a failure.

## 5. Why teams live on this (~10 min)

Pull the week together. The branch/PR workflow exists because of three properties you can now explain from the model:

- **Isolation.** A branch is a separate pointer, so work-in-progress lives off `main`. `main` stays trustworthy because broken or half-done commits aren't reachable from it until merged.
- **Review before merge.** The PR is a gate in front of the merge. The change is a *proposal* (commits on a branch) until someone approves it. Nothing lands on `main` unseen.
- **Parallel work without collisions.** Many people branch off `main` at once, each on their own pointer, and merge back. Git reconciles the lines; conflicts surface explicitly instead of one person silently clobbering another.

Tomorrow's CI robot bolts onto this: it watches for pushes and PRs (events on the graph you just learned) and runs your tests automatically *before* the merge gate opens. The model from today is what lets you reason about it.
