# Week 11, Day 4 — One Command to Deploy
*v1.0.0*

**Goal for today:** turn this week's pile of `aws` commands into a single, repeatable deploy. You'll write `deploy.sh` — build the site, then sync it to the bucket — so shipping an update is one command, not a remembered sequence. You'll also learn the `--delete` flag (so removed pages actually disappear from the live site) and set up the cost-and-tear-down discipline that keeps a real cloud account from quietly billing you.

## 2. Why a script (~10 min)

Right now, deploying an update means remembering: build the site, then run the right `aws s3 sync` with the right bucket name. Miss the build and you upload a stale site; fat-finger the bucket name and you upload nothing useful. That's the same fragility a hand-run sequence always has — and you've solved it before. In W8 a README's "usage" section captured the commands; here you go one better and put them *in a script* the computer runs, so the sequence can't be misremembered.

A **shell script** is just a file of terminal commands the computer runs top to bottom. You've used `flutter_with_secrets.sh`-style wrappers as a black box; today you write a small one yourself.

## 3. Write deploy.sh (~20 min)

```
subl deploy.sh
```

```bash
#!/usr/bin/env bash
set -euo pipefail

BUCKET="brand-lens-tyler-2026"          # <-- your bucket name

echo "Building site..."
python3 build_site.py

echo "Deploying to s3://$BUCKET ..."
aws s3 sync site/ "s3://$BUCKET" --delete

echo "Done. Live at:"
echo "http://$BUCKET.s3-website-us-east-1.amazonaws.com"
```

Read it line by line — a script you can't read is a command you can't trust (W4D1), and this one touches a billing account:

- **`#!/usr/bin/env bash`** — the "shebang": tells the system to run this file with `bash`. Standard first line of a shell script.
- **`set -euo pipefail`** — safety rails. `-e` stops the script the instant any command fails (so a failed build never proceeds to deploy a half-made site); `-u` errors on an undefined variable (a typo'd `$BUCKET` fails loudly instead of deploying to nowhere); `pipefail` makes a failure anywhere in a pipe count. Three words that turn "keeps going after an error" into "stops at the first sign of trouble" — exactly what you want when the next step publishes to the world.
- **`BUCKET="..."`** — your bucket name in one place. Change buckets? One edit. (Use *your* real name.)
- **`python3 build_site.py`** — rebuild first, always. The deploy can't ship a stale site because building *is* step one.
- **`aws s3 sync site/ "s3://$BUCKET" --delete`** — the upload, with the new flag below.
- The `echo` lines narrate progress and print the URL so you can click it.

### The `--delete` flag — why it matters

Plain `aws s3 sync` only *adds and updates*; it never removes. So if you delete a brand and rebuild, the brand's page vanishes from `site/` locally but **lingers in the bucket** — a ghost page, still live, now unreachable from the index but findable by its old URL. `--delete` fixes that: it makes the bucket *match* `site/` exactly, removing anything in the bucket that's no longer in your folder. The bucket becomes a mirror of your build, not an ever-growing pile. (Handle with the respect any "also deletes things" flag deserves: it deletes from the *bucket* to match your *local* folder, so always build before you sync — which the script guarantees by ordering.)

Make it executable and run it:

```
chmod +x deploy.sh
./deploy.sh
```

- **`chmod +x`** — marks the file runnable, so `./deploy.sh` works (one-time).

Watch it build, sync, and print the URL. Open the URL — same live site, shipped with *one command*. That's the deliverable: deploying is now `./deploy.sh`, full stop.

## 4. Prove --delete works (~10 min)

See the ghost-page fix for real. Pretend a brand left:

```
# temporarily remove a built page and re-sync WITHOUT --delete behavior first:
aws s3 ls s3://brand-lens-tyler-2026          # note the files up there
```

Now rebuild with one fewer brand (comment a brand out of `brands.yaml`, or just observe conceptually if you don't want to disturb data), run `./deploy.sh`, and `aws s3 ls` again — the dropped page is *gone* from the bucket, because `--delete` mirrored your folder. Restore `brands.yaml`, redeploy. The lesson sticks better having watched a file disappear from the cloud on command.

## 5. Cost and tear-down discipline (~15 min)

This is a real account with a real card attached, so build the habit now: **know what you're spending, and know how to make it stop.**

- **What this costs:** S3 charges tiny amounts for storage (your site is a few megabytes — fractions of a cent a month) and for requests (a few per visit). For a personal brief site this rounds to roughly nothing, and the free tier may cover it. It is *not* zero-attention, though — an abandoned account with forgotten resources is how people get surprise bills.
- **Set a guardrail:** in the AWS Console, **Billing → Budgets**, create a small monthly budget (say \$1) with an email alert. Now if anything ever costs more than expected, AWS emails you instead of you finding out on a statement. Cheap insurance.
- **Know the off switch.** When you're truly done with the site (end of course, or any time), tearing it down is two commands — empty the bucket, then remove it:

  ```
  aws s3 rm s3://brand-lens-tyler-2026 --recursive   # delete all objects
  aws s3 rb s3://brand-lens-tyler-2026               # remove the (now-empty) bucket
  ```

  `rb` ("remove bucket") refuses unless the bucket is empty — hence `rm --recursive` first. **Don't run these now** — you want the site up for Friday's demo and Week 12. Just *know* them. The ability to cleanly tear down what you stood up is part of being trusted with infrastructure.

## 6. Commit (~5 min)

`deploy.sh` is a real project artifact — ship it through the workflow:

```
git switch -c add-deploy-script
git add deploy.sh
git commit -m "Add deploy.sh: build the site and sync to S3 with --delete"
git push -u origin add-deploy-script
```

Open the PR, watch CI go green, merge, delete the branch, `git pull`. Note what CI does and doesn't do here: it runs your `pytest` suite (good), but it does *not* run `deploy.sh` — the robot has no AWS key and shouldn't deploy from a random PR. *Who* gets to deploy, and *when*, is exactly the question Week 12 answers.
