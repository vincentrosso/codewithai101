## 2. What a "unit" is (~10 min)

A **unit test** tests a **unit**. The word "unit" is vaguer than it sounds. People use it to mean different things, but here's the working definition for this course:

> A unit is the smallest piece of your code that does one thing you can describe in a sentence.

For `brand-lens`, the units are functions. `parse_page(html, url) -> dict` is a unit — "given HTML and a URL, produce a record dict." `fetch_one(url) -> (status, html, error)` is a unit. `process_url(url) -> dict` is a unit, though a chunkier one because it calls both of the above.

What's **not** a unit:

- The whole `runner.py main()` — that loads config, loops over brands, fetches pages, parses them, writes JSON. Too many things in one place to verify in isolation.
- "The pipeline" — that's the system, not a unit.
- A single line of code — too small. You can't say in a sentence what one line does without context.

The instinct to build up to: when you write a function, ask *what would a test for this look like?* If the answer is hard, the function probably does too much. Splitting it makes it both testable and clearer.

## 3. Pure vs side-effectful functions (~10 min)

`parse_page` was easy to test on D1 and D2. `fetch_one` would not have been. The difference has a name.

A function is **pure** if:

1. Given the same input, it always returns the same output.
2. It doesn't change anything outside itself — no writing files, no sending network requests, no modifying global state.

`parse_page(html, url)` is pure. Hand it the same HTML string, you get the same dict every time. It doesn't touch the network, doesn't write a file, doesn't change a global variable. That's why testing it was easy: you wrote the input as a Python string in the test, called the function, asserted on the output. Done.

A function has a **side effect** if it does anything outside returning a value:

- `fetch_one("https://example.com")` — sends an HTTP request. Side effect.
- A function that writes to `results.json` — file I/O. Side effect.
- A function that prints to the terminal — also a side effect, though a mild one.
- A function that reads the current time — depends on outside state. Hard to test because the time keeps changing.

Side-effectful functions are harder to test because the test has to **fake the side effect**. You can't actually fetch from the internet in a unit test — too slow, too unreliable, depends on what's running on a server you don't control. So you substitute a fake version of `requests.get` during the test. That technique is called **mocking**, and it's the entire subject of tomorrow's lesson.

The takeaway for today: **pure functions are easy to test, side-effectful functions are hard.** A codebase that pushes side effects to the edges (fetch, write file, call API) and keeps the interior pure (parse, transform, decide) is dramatically easier to test. `brand-lens` already has this shape — that's not an accident. W3D2 split the modules along exactly this line: `fetch.py` (side effects) and `parse.py` (pure) are deliberately separated.

## 4. The test pyramid (~10 min)

When people draw a picture of a test suite, it usually looks like this:

```
          /\
         /e2\         <- few, slow, brittle
        /----\
       /integ.\       <- some, medium, real components together
      /--------\
     /  unit    \     <- many, fast, isolated
    /------------\
```

Three layers:

- **Unit tests** at the base. Each tests one function (one unit) in isolation. Cheap to write, fast to run (milliseconds), don't touch the network or filesystem. You should have many of them. Everything you wrote D1 and D2 is a unit test.

- **Integration tests** in the middle. Test that several pieces *fit together*. Example: `process_url(url)` calls `fetch_one` then `parse_page` — does the result come out shaped correctly? Slower (might touch the network), fewer in number. You haven't written any of these yet.

- **End-to-end tests** at the top. Run the whole system from outside, top to bottom. Example: invoke `python3 runner.py --input test_brands.yaml --output /tmp/out.json` and check `/tmp/out.json` looks right. Very slow, very brittle (anything anywhere in the pipeline breaks the test), few in number.

The shape is a *pyramid*, not a rectangle, on purpose: most of your tests should be at the bottom, because unit tests give you the fastest feedback for the lowest cost. End-to-end tests are valuable but expensive — one or two confirm the whole machine works; you don't need fifty.

A common anti-pattern is the **ice cream cone** — many e2e tests, few unit tests. It feels comprehensive ("we test the whole thing!") but it's slow, hard to debug (when an e2e test fails, *where* did it break?), and flaky.

For now, Week 5 is unit tests only. Integration tests will probably show up later in the course.

## 5. Tests as a spec (~5 min)

This one came up on D2 — worth restating because it's the deepest idea in testing.

A function's docstring tells you what the author *thought* the function did. A test tells you what the function *actually* does, today, on this machine. If the test passes, the behavior is real. If you delete the function and rewrite it, the tests still verify the same promise — they're independent of the implementation.

This is why people say **"if it's not tested, it doesn't work."** That's a slight exaggeration, but the seed of truth is: code without tests has no automated way to detect when it stopped doing what it was supposed to do. It works *right now, in the way you happen to be using it*. That's a weaker guarantee than it sounds.

## 6. What to do when a test fails (~10 min)

A failing test is a question: **is the code wrong, or is the test wrong?**

Most of the time, the code is wrong — you changed something, you broke a behavior the test was protecting. The fix is in the code, not the test.

Sometimes the test is wrong — the *requirement* changed. The function used to return a string, now it returns a dict. The old test asserted on a string. The test isn't a bug; it just describes the old behavior. The fix is to update the test (and the docstring, if there is one) to describe the new behavior.

The dangerous move is to fix a failing test by relaxing its assertion until it passes ("oh, I guess `None` is also fine here"). That deletes the alarm without fixing the alarm's cause. If you find yourself softening a test, stop and ask: *am I changing this because the requirement actually changed, or because I don't want to deal with the failure?*

A rule of thumb: when a test fails, **read the failure message first, before touching anything**. The message tells you what the test expected vs. what it got. That comparison usually points you at the right answer — code or test — without guessing.

## 7. Flaky tests (~5 min)

A **flaky test** passes sometimes and fails sometimes, with no code change in between. It's the worst kind of test, because it erodes trust in the whole suite. After a few flaky failures, people start ignoring red — and then a real failure also gets ignored.

Common causes:

- **Hidden time dependency.** The test compares against `datetime.now()` and the clock moved.
- **Hidden network dependency.** The test fetches a URL "just to check" and the URL was down today.
- **Hidden ordering dependency.** Test A leaves a file behind that test B relies on. Run them in a different order, B fails.
- **Hidden randomness.** The test uses a random sample and the assertion only holds for *most* random samples.

The fix is almost always to remove the hidden dependency — fake the clock, mock the network, isolate the file. You'll meet the first version of this tomorrow when you mock `requests.get`.
