## 2. What you're about to do (~5 min)

An **API** (Application Programming Interface) is a way for one program to ask another program — usually over the internet — to do something. Anthropic runs Claude on their servers. They expose an API: your program sends an HTTP request containing your prompt, their server runs the model, and sends back the response. It's the same request/response shape as `requests.get` from Week 1 (W2D3) — you're just talking to Claude's servers instead of a brand's website, and the `anthropic` package handles the HTTP details for you.

Two things make it different from fetching a web page:

- **It costs money.** Each call is billed by the amount of text in and out (D3 explains "tokens"). The calls you make this week cost fractions of a cent each, but the *key* that authorizes them is tied to a real account with real money.
- **It needs a secret key.** The API has to know it's you. That key is like a password. Anyone who gets it can spend your money. So the entire first half of today is about handling that key correctly — getting it, and making sure it never, ever lands in git.

## 3. Get an API key (~10 min)

1. Go to **console.anthropic.com** and sign in (or have Vincent set you up with access). You may need to add a small amount of credit to the account.
2. Find **API Keys** and create a new key. It looks like `sk-ant-...` — a long string.
3. **Copy it once, now.** The console shows it a single time. If you lose it, you make a new one; you can't view the old one again.

Do **not** paste it into a chat, a file you'll commit, or a screenshot. Treat it like the password to a bank account, because functionally it is one.

## 4. Store the key safely with a .env file (~15 min)

You never put secrets in code. The standard pattern — the same one mkecitysmart and the autoarb scraper use — is a `.env` file that holds secrets, is read at runtime, and is **never committed**.

First, make sure git ignores it. Open `.gitignore` (you made one on W6D1) and add:

```
.env
```

**Do this before creating the .env file**, so there's no window where it's trackable.

Now create the file:

```
touch .env
subl .env
```

Put your key in it — one line, no quotes, no spaces around the `=`:

```
ANTHROPIC_API_KEY=sk-ant-your-actual-key-here
```

Save. Then confirm git is ignoring it:

```
git status
```

`.env` should **not** appear in the list of changes. If it does, your `.gitignore` line is wrong — fix it before going further. This is the single most important check of the day.

It's good practice to commit a *template* that shows what the file should contain, without the secret:

```
touch .env.example
subl .env.example
```

```
ANTHROPIC_API_KEY=sk-ant-...
```

`.env.example` is safe to commit — it has no real key, just the shape. Anyone setting up the project later copies it to `.env` and fills in their own.

## 5. Install the packages (~5 min)

```
pip install anthropic python-dotenv
```

- **`anthropic`** — Anthropic's official Python SDK. It wraps the HTTP calls so you write Python, not raw web requests.
- **`python-dotenv`** — reads your `.env` file and loads the values into the environment when your program starts. Without it, your program wouldn't see `ANTHROPIC_API_KEY`.

If you keep a `requirements.txt`, add both.

## 6. Your first call (~20 min)

```
touch first_call.py
subl first_call.py
```

```python
from dotenv import load_dotenv
from anthropic import Anthropic

load_dotenv()                 # read .env into the environment
client = Anthropic()          # reads ANTHROPIC_API_KEY automatically

message = client.messages.create(
    model="claude-haiku-4-5",
    max_tokens=300,
    messages=[
        {"role": "user", "content": "In one sentence, what is a brand's value proposition?"}
    ],
)

print(message.content[0].text)
```

Read every line:

- **`load_dotenv()`** — reads `.env` and makes `ANTHROPIC_API_KEY` available. Must run *before* you create the client.
- **`Anthropic()`** — the client object. With no arguments, it looks for `ANTHROPIC_API_KEY` in the environment (which `load_dotenv` just put there). You never write the key in the code — the client finds it.
- **`model="claude-haiku-4-5"`** — *which* Claude. Haiku is the smallest, fastest, cheapest model. For learning and for short brand summaries it's plenty, and it keeps the bill near zero. (Bigger models exist — Sonnet, Opus — for harder work. You'd change one string.)
- **`max_tokens=300`** — the most text Claude is allowed to generate in the reply. A safety cap: it stops a runaway response from costing more than you intended. (D3: a token is roughly ¾ of a word.)
- **`messages=[{"role": "user", "content": "..."}]`** — the conversation, as a list. One message here, from `"user"` (you). The model replies as the `"assistant"`. This is the same chat shape you've used in the browser, written as data.
- **`message.content[0].text`** — digging the text out of the response. The reply is an object; `.content` is a list of content blocks; `[0]` is the first block; `.text` is its text. You'll meet this shape again, so don't gloss over it.

**Predict before you run.** Write in your journal, in one sentence, what you expect Claude to say. Then:

```
python3 first_call.py
```

You should see a one-sentence answer about value propositions. **It cost you a fraction of a cent and you just used Claude from inside a program for the first time.**

## 7. Run it again — and notice something (~5 min)

Run it a second time:

```
python3 first_call.py
```

The answer is *similar but not identical* to the first run. Same question, different wording. This is important and it's nothing like the code you've written all course: `parse_page` returns the exact same dict every time for the same input (it's *pure*, W5D3). An LLM call does not. Given the same prompt, you get a *different* response each time.

If your section-6 prediction assumed you could guess the exact wording, edit your journal: **you can't.** This non-determinism is the single biggest thing that makes LLM calls different to program around, and it's why testing them (D4) needs a special trick. Note in your journal how the two runs differed.
